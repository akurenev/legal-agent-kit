## 2. Термины

2.1. **Сервис** — сайт, приложение, PWA, личный кабинет, бот, API и иные компоненты **[НАЗВАНИЕ СЕРВИСА]**.

2.2. **Пользователь** — лицо, которое использует сервис и подключает внешние интеграции.

2.3. **Интеграция** — подключение сервиса к внешнему API, маркетплейсу, CRM, платежному сервису, мессенджеру, поставщику данных или иному внешнему сервису.

2.4. **API-токен / ключ доступа** — API key, Bearer token, OAuth token, refresh token, пароль приложения, webhook secret или иной идентификатор, который используется для подключения к внешнему сервису или API.

2.5. **OAuth-доступ** — механизм авторизации, при котором пользователь разрешает сервису получать доступ к определенным данным или действиям во внешнем сервисе.

2.6. **Webhook** — механизм отправки событий из сервиса на указанный пользователем URL или получения событий от внешнего сервиса.

2.7. **Внешний сервис** — сервис третьего лица, к которому подключается **[НАЗВАНИЕ СЕРВИСА]** для получения, передачи или обработки данных.

2.8. **Данные интеграции** — данные, полученные или переданные через внешнюю интеграцию, включая заказы, товары, остатки, сообщения, операции, статусы, события, логи и другие данные.

## 7. OAuth-подключения

7.1. Если интеграция подключается через OAuth, пользователь может быть перенаправлен на страницу внешнего сервиса для авторизации и выдачи разрешений.

7.2. Внешний сервис может показывать пользователю список запрашиваемых прав доступа. Пользователь самостоятельно принимает решение о предоставлении такого доступа.

7.3. После успешного OAuth-подключения сервис может получить access token, refresh token, идентификатор аккаунта во внешнем сервисе, список разрешений и иные данные, предоставленные внешним сервисом.

7.4. Сервис использует OAuth-доступ только для целей, связанных с выбранной пользователем интеграцией.

7.5. Пользователь может отключить OAuth-интеграцию в сервисе, если такая функция предусмотрена, а также во внешнем сервисе, если он предоставляет такую возможность.

7.6. Если внешний сервис отзывает OAuth-доступ, изменяет разрешения или блокирует аккаунт пользователя, интеграция может перестать работать.

## 8. Хранение токенов

8.1. Сервис хранит API-токены и ключи доступа в объеме, необходимом для работы выбранных пользователем интеграций.

8.2. Сервис принимает разумные технические и организационные меры для защиты токенов от неправомерного доступа, раскрытия, изменения или удаления.

8.3. Такие меры могут включать:

- ограничение доступа к токенам;
- разграничение прав сотрудников и сервисов;
- шифрование токенов;
- хранение секретов вне публичного репозитория;
- маскирование токенов в интерфейсе;
- исключение токенов из логов;
- контроль доступа к административным функциям;
- аудит действий с интеграциями;
- резервное копирование и мониторинг безопасности.

8.4. Конкретный способ хранения токенов: [СПОСОБ ХРАНЕНИЯ ТОКЕНОВ / УТОЧНИТЬ].

8.5. Если токены хранятся в зашифрованном виде, способ шифрования, управление ключами и доступ к ключам должны быть описаны во внутренней технической документации и не должны раскрываться в публичных документах сверх необходимого.

8.6. Токены не должны отображаться пользователю полностью после сохранения. В интерфейсе рекомендуется показывать только маску, например:

```text
abcd********wxyz
```

8.7. Токены не должны передаваться через GET-параметры URL, если это не является частью безопасного и обоснованного протокола.

## 9. Использование токенов

9.1. Сервис использует токены только для выполнения функций, связанных с подключенной интеграцией.

9.2. В зависимости от интеграции сервис может выполнять следующие действия:

- читать данные внешнего сервиса;
- передавать данные во внешний сервис;
- обновлять статусы;
- отправлять события;
- получать события;
- проверять доступность API;
- получать справочную информацию;
- выполнять действия, явно выбранные пользователем в интерфейсе;
- выполнять автоматические действия, предусмотренные настройками пользователя, тарифом или документацией.

9.3. Сервис не должен использовать токены для действий, которые не относятся к заявленной функциональности интеграции.

9.4. Если внешний сервис ограничивает количество запросов, права доступа или набор функций, работа интеграции может быть ограничена.

9.5. Если токен становится недействительным, истекает, отзывается или не имеет нужных прав, сервис может приостановить интеграцию и уведомить пользователя доступным способом.

15.1. Внешние сервисы могут быть недоступны, работать с ошибками, менять API, ограничивать запросы, отзывать токены, блокировать аккаунты или менять правила использования.

15.2. Владелец сервиса не отвечает за действия, бездействие, ошибки, недоступность, ограничения, блокировки или изменения правил внешних сервисов, если они находятся вне контроля владельца сервиса.

15.3. При ошибке внешней интеграции сервис может:

- повторить запрос;
- отложить обработку;
- пометить интеграцию как ошибочную;
- уведомить пользователя;
- отключить интеграцию;
- запросить обновление токена;
- обратиться к пользователю за уточнением настроек.

15.4. Пользователь понимает, что работа интеграций зависит не только от сервиса, но и от внешних API, прав доступа, лимитов, состояния аккаунта пользователя и качества переданных настроек.

## 16. Безопасность пользователя

Пользователь обязан:

- хранить свои API-токены и ключи доступа в безопасности;
- не передавать токены третьим лицам без необходимости;
- использовать токены с минимально необходимыми правами, если внешний сервис это позволяет;
- своевременно отзывать ненужные токены;
- обновлять токены при подозрении на компрометацию;
- не передавать токены через незащищенные каналы;
- не публиковать токены в открытых репозиториях, чатах, скриншотах и документах;
- контролировать действия сотрудников, имеющих доступ к внешним сервисам.
