# Codex Prompt: Integrate Legal Pages

## Назначение

Этот промпт используется для Codex, когда правовые документы уже подготовлены и их нужно аккуратно встроить в сайт, SaaS-сервис, PWA, личный кабинет, Telegram/MAX-бот или API-документацию.

Промпт не предназначен для изменения бизнес-логики проекта. Его цель — добавить публичные страницы документов, ссылки, чекбоксы согласий и рекомендации по хранению фактов принятия документов.

---

## Основной промпт

```text
Нужно внедрить правовые документы в интерфейс проекта.

Правовые документы уже подготовлены в:

/docs/legal/generated/

Используй Legal Agent Kit как методическую базу.

Если Legal Agent Kit подключен локально, используй:
- /docs/legal-kit/AGENTS.md
- /docs/legal-kit/LEGAL_AGENT_GUIDE.md
- /docs/legal-kit/docs/interface-checklist.md
- /docs/legal-kit/docs/consent-storage.md
- /docs/legal-kit/docs/document-matrix.md

Если Legal Agent Kit не подключен локально, используй публичный репозиторий:
[ССЫЛКА НА LEGAL AGENT KIT]

---

## Цель задачи

Аккуратно добавить правовые документы в проект:

1. Создать публичные страницы документов.
2. Добавить ссылки на документы в интерфейс.
3. Добавить чекбоксы согласий там, где они нужны.
4. Подготовить или предложить механизм хранения фактов принятия документов.
5. Не менять бизнес-логику проекта.
6. Не ломать существующий дизайн и маршруты.

---

## Важное ограничение

На первом этапе не вноси изменения в код.

Сначала выполни анализ проекта и подготовь план внедрения:

- какие файлы нужно изменить;
- какие страницы нужно создать;
- какие ссылки добавить;
- какие чекбоксы добавить;
- какие согласия нужно фиксировать;
- какие риски есть;
- какие вопросы нужно уточнить.

Только после отдельного подтверждения можно готовить патч.

## Этап 4. Определи, куда добавить ссылки

Проверь необходимость добавления ссылок в следующие места:

### Футер сайта

Добавить ссылки на актуальные документы:

- Пользовательское соглашение;
- Политика конфиденциальности;
- Согласие на обработку персональных данных;
- Публичная оферта;
- Правила оплаты и возврата;
- Правила подписки;
- Политика cookies;
- Правила поддержки;
- Контакты.

Набор ссылок зависит от документов, которые реально есть в проекте.

### Регистрация

Если есть регистрация, добавить ссылки и чекбоксы:

```text
[ ] Я принимаю Пользовательское соглашение
[ ] Я ознакомлен с Политикой конфиденциальности
[ ] Я даю согласие на обработку персональных данных
```

Если есть рекламные сообщения:

```text
[ ] Я согласен получать информационные и рекламные сообщения
```

Согласие на рекламные сообщения не должно быть обязательным, если оно не является необходимым для оказания услуги.

### Форма обратной связи

Если есть форма заявки или обратной связи, добавить согласие:

```text
[ ] Я даю согласие на обработку персональных данных для обработки моего обращения
```

Или текст под формой:

```text
Нажимая «Отправить», я даю согласие на обработку персональных данных и соглашаюсь с Политикой конфиденциальности.
```

### Страница оплаты

Если есть оплата, добавить текст:

```text
Нажимая «Оплатить», я принимаю условия Публичной оферты и Правила оплаты и возврата.
```

Если есть подписка:

```text
Нажимая «Оплатить», я принимаю условия Публичной оферты, Правила подписки и Правила оплаты и возврата.
```

Если есть автопродление, явно указать:

```text
Подписка продлевается автоматически до ее отключения пользователем. Условия автопродления описаны в Правилах подписки.
```

### Страница тарифов

Добавить ссылки на:

- Публичную оферту;
- Правила подписки;
- Правила оплаты и возврата.

Проверить, что тарифы в интерфейсе не противоречат документам.

### Личный кабинет

Добавить раздел или ссылки:

```text
Правовые документы
```

Возможные элементы:

- актуальные документы;
- дата редакции;
- версия документа;
- история принятых согласий, если она хранится;
- ссылки на отключение рассылок;
- ссылки на удаление аккаунта или запрос удаления данных, если предусмотрено проектом.

### Настройки интеграций

Если есть API-токены, OAuth, webhook, маркетплейсы или CRM, добавить предупреждение и ссылку на документ:

```text
Передавая API-токен, вы разрешаете сервису использовать его для выполнения выбранных функций интеграции. Вы можете удалить токен в настройках интеграции.
```

Ссылка:

```text
Правила обработки API-токенов и данных интеграций
```

### Подключение банка

Если есть подключение банка, добавить предупреждение и ссылку:

```text
Сервис получает данные по счетам только для отображения уведомлений, сводок и аналитики. Сервис не инициирует платежи и не распоряжается денежными средствами, если иное явно не указано в условиях сервиса.
```

Ссылка:

```text
Правила обработки банковских данных
```

### Telegram/MAX-бот

Если есть бот, предложить сообщение первого запуска или кнопку принятия условий:

```text
Перед использованием сервиса ознакомьтесь с Пользовательским соглашением и Политикой конфиденциальности. Продолжая работу с ботом, вы подтверждаете согласие с условиями сервиса.
```

Для более строгого сценария:

```text
[Принять условия]
```

## Этап 7. После подтверждения подготовь патч

Если пользователь подтвердил внесение изменений, подготовь патч.

Правила:

1. Не меняй бизнес-логику.
2. Не меняй платежную логику.
3. Не меняй регистрацию глубже, чем требуется для чекбоксов и ссылок.
4. Не меняй API-контракты без необходимости.
5. Не удаляй существующие документы.
6. Не ломай существующие URL.
7. Не меняй дизайн радикально.
8. Следуй текущему стилю проекта.
9. Разделяй изменения по смыслу.
10. Добавляй комментарии только там, где они помогают понять логику.

## Этап 11. Проверка результата

После внесения изменений проверь:

- страницы документов открываются;
- ссылки в футере работают;
- ссылки на регистрации работают;
- ссылки на оплате работают;
- документы доступны без авторизации;
- мобильная верстка не сломана;
- чекбоксы не отмечены заранее;
- без обязательных согласий регистрация невозможна, если это предусмотрено логикой;
- рекламное согласие не является обязательным, если оно не нужно для услуги;
- принятие оферты фиксируется или есть план фиксации;
- нет ошибок сборки frontend;
- нет ошибок backend;
- существующие тесты проходят, если они есть.
