
# AGENTS.md

## Назначение репозитория

**Legal Agent Kit** — открытый набор инструкций, шаблонов, чек-листов, примеров и готовых промптов для подготовки черновиков правовых документов цифрового проекта.

Репозиторий предназначен для использования:

- AI-агентами;
- Codex;
- ChatGPT;
- разработчиками;
- предпринимателями;
- продуктовыми командами;
- владельцами сайтов, SaaS-сервисов, PWA, ботов и API-сервисов.

Агент должен использовать этот репозиторий как методическую базу, а не как источник универсальных финальных юридических документов.

---

## Главный принцип работы

Агент не должен сразу писать «готовые документы».

Сначала агент должен:

1. Понять фактическую модель проекта.
2. Заполнить анкету проекта.
3. Определить нужный комплект документов.
4. Выявить неизвестные данные.
5. Подготовить вопросы к владельцу проекта.
6. Только после этого готовить черновики документов.

Документы должны быть адаптированы под фактическую работу проекта, а не под абстрактный тип сайта.

---

## Основная задача агента

Агент помогает подготовить комплект правовых документов для:

- простого лендинга;
- сайта-визитки;
- корпоративного сайта;
- SaaS-сервиса;
- PWA-приложения;
- Telegram/MAX-бота;
- API-сервиса;
- сервиса подписки;
- маркетплейс-интеграции;
- сервиса с оплатой;
- сервиса с банковскими данными;
- сервиса с внешними интеграциями;
- другого цифрового продукта.

Результатом работы должны быть:

- заполненная анкета проекта;
- список необходимых документов;
- матрица документов;
- черновики документов в Markdown;
- список неизвестных данных и плейсхолдеров;
- список вопросов к владельцу проекта или юристу;
- список юридических рисков;
- рекомендации по внедрению документов в интерфейс;
- рекомендации по хранению фактов согласия пользователей;
- рекомендации по обновлению документов при изменении функциональности.

## Обязательный порядок работы

Перед созданием документов агент должен:

1. Изучить описание проекта пользователя.
2. Изучить доступные файлы проекта, если они предоставлены:
   - `README.md`;
   - `AGENTS.md`;
   - `.env.example`;
   - маршруты API;
   - страницы регистрации;
   - страницы оплаты;
   - страницы тарифов;
   - личный кабинет;
   - настройки интеграций;
   - футер сайта;
   - тексты тарифов;
   - существующие правовые документы;
   - документацию API;
   - конфигурацию аналитики, cookies и платежей.
3. Определить тип проекта:
   - лендинг;
   - сайт;
   - SaaS;
   - PWA;
   - Telegram/MAX-бот;
   - интернет-магазин;
   - API-сервис;
   - сервис подписки;
   - маркетплейс-интеграция;
   - сервис обработки данных;
   - сервис с банковскими или платежными интеграциями.
4. Заполнить `LEGAL_PROJECT_QUESTIONNAIRE.md` или создать файл анкеты проекта.
5. Выбрать комплект документов по `LEGAL_AGENT_GUIDE.md` и `docs/document-matrix.md`.
6. Подготовить матрицу документов.
7. Подготовить черновики документов на основе шаблонов из `templates/ru/`.
8. Оставить плейсхолдеры для неизвестных данных.
9. Подготовить список вопросов для владельца проекта или юриста.
10. Подготовить список юридических рисков.
11. Подготовить рекомендации по внедрению документов в интерфейс.
12. Подготовить рекомендации по хранению пользовательских согласий.
13. Отметить, что документы требуют юридической проверки перед публикацией.

## Что агент должен учитывать

При подготовке документов агент должен учитывать:

- юрисдикцию проекта;
- страну регистрации владельца сервиса;
- страну нахождения пользователей;
- модель проекта: B2B, B2C или смешанная;
- наличие регистрации и личного кабинета;
- наличие ролей пользователей;
- наличие платежей, подписок, пробного периода и возвратов;
- наличие автопродления;
- наличие льготного периода;
- состав собираемых персональных данных;
- цели обработки данных;
- сроки хранения данных;
- использование cookies, аналитики и рекламных систем;
- наличие рекламных сообщений и рассылок;
- наличие API, webhook, токенов, ключей доступа и интеграций;
- наличие банковских данных или данных, полученных через авторизацию в банке;
- наличие Telegram/MAX-бота или другого мессенджера;
- наличие пользовательского контента, сообщений, файлов и вложений;
- наличие внешних поставщиков, маркетплейсов, CRM, банков и платежных сервисов;
- фактическую работу проекта, а не только маркетинговое описание.

- Правила обработки API-токенов и данных интеграций;
- Правила обработки банковских данных;
- Условия использования API;
- Правила использования бота;
- список юридических рисков;
- список вопросов для владельца проекта;
- чек-лист внедрения правовых документов в интерфейс;
- план обновления документов при изменении функциональности.

## Как выбирать документы

Агент должен использовать `docs/document-matrix.md` и учитывать следующие базовые правила:

| Условие проекта | Обычно нужны документы |
|---|---|
| Есть форма заявки | `privacy.md`, `personal_data_consent.md` |
| Есть cookies или аналитика | `cookie_policy.md`, раздел в `privacy.md` |
| Есть регистрация | `terms.md`, `privacy.md`, `personal_data_consent.md` |
| Есть личный кабинет | `terms.md`, `privacy.md` |
| Есть онлайн-оплата | `offer.md`, `payment_and_refund.md`, `privacy.md` |
| Есть подписка, trial или льготный период | `subscription_rules.md`, `offer.md`, `payment_and_refund.md` |
| Есть рекламные сообщения | `marketing_consent.md`, раздел в `privacy.md` |
| Есть API или webhook для клиентов | `api_terms.md`, `data_processing.md` |
| Пользователь передает API-токены | `data_processing.md`, раздел в `privacy.md` |
| Есть банковские данные | `bank_data_processing.md`, `privacy.md`, `personal_data_consent.md` |
| Есть Telegram/MAX-бот | `bot_terms.md`, `privacy.md`, `personal_data_consent.md` |
| Есть поддержка с условиями | `support_rules.md` |

## Требования к стилю документов

Документы должны быть:

- понятными для обычного пользователя;
- структурированными;
- адаптированными под конкретный проект;
- без избыточной юридической сложности;
- с едиными терминами во всех документах;
- с нумерацией разделов;
- с датой редакции и номером версии;
- с плейсхолдерами для неизвестных данных;
- без универсальных фраз, которые не относятся к проекту;
- без противоречий между документами, тарифами и интерфейсом.

Рекомендуемый верхний блок документа:

```text
Редакция от: [ДАТА]
Версия: [ВЕРСИЯ]
Сервис: [НАЗВАНИЕ СЕРВИСА]
Домен: [ДОМЕН]
```

## Плейсхолдеры

Если данных недостаточно, агент не должен выдумывать значения.

Нужно использовать плейсхолдеры:

```text
[НАЗВАНИЕ СЕРВИСА]
[ДОМЕН]
[ВЛАДЕЛЕЦ СЕРВИСА]
[ОРГАНИЗАЦИОННО-ПРАВОВАЯ ФОРМА]
[ИНН]
[ОГРН/ОГРНИП]
[АДРЕС]
[EMAIL]
[ТЕЛЕФОН]
[ПЛАТЕЖНЫЙ ПРОВАЙДЕР]
[УСЛОВИЯ ВОЗВРАТА]
[СПИСОК ПЕРСОНАЛЬНЫХ ДАННЫХ]
[СПИСОК ИНТЕГРАЦИЙ]
[СРОК ХРАНЕНИЯ ДАННЫХ]
[ЮРИСДИКЦИЯ]
[УТОЧНИТЬ]
```

После подготовки документов агент должен отдельно вывести список всех оставленных плейсхолдеров.

---

## Что запрещено агенту

Агент не должен:

- выдавать документы как юридически финальные;
- обещать соответствие законодательству без проверки юристом;
- выдумывать реквизиты компании, адреса, ИНН, ОГРН/ОГРНИП, тарифы, условия возврата или платежных провайдеров;
- копировать документы с чужих сайтов;
- удалять важные ограничения ответственности;
- смешивать согласие на обработку персональных данных с пользовательским соглашением;
- делать согласие на рекламные сообщения обязательным без необходимости;
- скрывать неизвестные данные вместо использования плейсхолдеров;
- игнорировать фактическую работу сервиса;
- менять смысл тарифов, оплаты, возвратов или подписки без подтверждения владельца проекта;
- изменять рабочий код проекта без отдельного указания пользователя;
- публиковать документы без явного подтверждения пользователя.

## Требования к внедрению в интерфейс

Если агенту поручено не только подготовить документы, но и предложить внедрение, он должен проверить наличие:

- ссылок на документы в футере сайта;
- ссылок на документы на странице регистрации;
- ссылок на документы на странице оплаты;
- ссылок на документы в Telegram/MAX-боте;
- отдельных чекбоксов согласий;
- текста согласия перед оплатой;
- отдельного согласия на рекламные сообщения;
- страницы со списком актуальных документов;
- механизма хранения факта принятия документов;
- версии документа и даты принятия согласия;
- контакта для вопросов по персональным данным;
- контакта для вопросов по оплате и возвратам;
- механизма отзыва необязательных согласий.

Пример чекбоксов для регистрации:

```text
[ ] Я принимаю Пользовательское соглашение
[ ] Я ознакомлен с Политикой конфиденциальности
[ ] Я даю согласие на обработку персональных данных
```

Пример необязательного рекламного согласия:

```text
[ ] Я согласен получать информационные и рекламные сообщения
```

Пример текста перед оплатой:

```text
Нажимая «Оплатить», я принимаю условия Публичной оферты и Правила оплаты и возврата.
```

Пример текста для бота:

```text
Перед использованием сервиса ознакомьтесь с Пользовательским соглашением и Политикой конфиденциальности. Продолжая работу с ботом, вы подтверждаете согласие с условиями сервиса.
```

## Хранение согласий

Если в проекте есть регистрация, личный кабинет, подписка, оплата, бот, API, интеграции или формы заявок, агент должен предложить хранить факты принятия документов.

Рекомендуемые данные для хранения:

- ID пользователя;
- ID аккаунта или организации, если есть;
- тип документа;
- название документа;
- версия документа;
- дата редакции документа;
- URL документа;
- текст чекбокса или подтверждения;
- тип события: `accepted` или `revoked`;
- дата и время принятия;
- дата и время отзыва, если применимо;
- IP-адрес;
- user-agent;
- источник согласия;
- язык интерфейса;
- дополнительные метаданные.

Примеры типов документов:

```text
terms
privacy
personal_data_consent
offer
payment_and_refund
subscription_rules
marketing_consent
cookie_policy
data_processing
bank_data_processing
api_terms
bot_terms
support_rules
```

## Финальная проверка

Перед завершением работы агент должен проверить:

- совпадает ли название сервиса во всех документах;
- совпадают ли реквизиты владельца;
- указана ли дата редакции;
- указана ли версия документов;
- нет ли противоречий между офертой, тарифами и правилами оплаты;
- описаны ли условия возврата;
- описаны ли платежные провайдеры;
- описаны ли персональные данные и цели их обработки;
- описаны ли сроки хранения данных;
- описаны ли cookies, аналитика и рекламные пиксели;
- описаны ли внешние сервисы и интеграции;
- описаны ли API-токены, ключи доступа и банковские данные, если они используются;
- описаны ли Telegram/MAX ID, сообщения и уведомления, если есть бот;
- выделено ли отдельное согласие на обработку персональных данных;
- выделено ли отдельное согласие на рекламные сообщения, если они есть;
- подготовлены ли тексты чекбоксов;
- подготовлен ли список плейсхолдеров;
- подготовлены ли вопросы для владельца проекта или юриста;
- подготовлен ли список юридических рисков;
- подготовлен ли чек-лист внедрения документов в интерфейс.
