

# Чек-лист API-токенов и внешних интеграций

## Назначение

Этот чек-лист помогает проверить, что проект корректно описывает получение, хранение, использование и удаление API-токенов, OAuth-доступов, webhook-адресов и данных внешних интеграций.

Чек-лист используется для SaaS-сервисов, API-сервисов, маркетплейс-интеграций, банковских интеграций, CRM-интеграций, сервисов уведомлений, PWA и ботов, где пользователь подключает внешние сервисы или передает ключи доступа.

Чек-лист не заменяет юридическую проверку. Он помогает найти технические и продуктовые риски, которые должны быть отражены в документах и интерфейсе.

---

## 1. Когда использовать чек-лист

Используйте этот чек-лист, если проект работает с:

- API-ключами пользователей;
- Bearer-токенами;
- OAuth access token;
- OAuth refresh token;
- webhook URL клиента;
- secret для подписи webhook;
- ключами маркетплейсов;
- токенами банков;
- токенами CRM;
- токенами мессенджеров;
- внешними поставщиками данных;
- интеграциями с сервисами аналитики;
- интеграциями с платежными сервисами;
- сервисными API между внутренними модулями.

## 5. Получение токена от пользователя

Если пользователь вводит токен вручную или подключает внешний сервис, интерфейс должен объяснять, что происходит.

| Проверка | Статус | Комментарий |
|---|---|---|
| Пользователь понимает, какой сервис подключает | [ ] | Например: Ozon, Яндекс Маркет, банк, CRM |
| Пользователь понимает, какие данные будут доступны сервису | [ ] | Заказы, товары, операции, остатки, сообщения |
| Пользователь понимает, какие действия сервис сможет выполнять | [ ] | Читать данные, отправлять события, обновлять остатки |
| Есть ссылка на правила обработки токенов | [ ] | `data_processing.md` |
| Есть ссылка на Политику конфиденциальности | [ ] | Если через токен получаются данные |
| Есть предупреждение об ответственности за корректность токена | [ ] | Пользователь вводит актуальный и разрешенный токен |
| Есть возможность удалить токен | [ ] | В настройках интеграции или через поддержку |
| Есть возможность отключить интеграцию | [ ] | Желательно |
| Токен не отображается полностью после сохранения | [ ] | Показывать только маску |
| Токен не попадает в URL | [ ] | Не передавать токены через GET-параметры |

Пример текста:

```text
Передавая API-токен, вы разрешаете сервису использовать его для выполнения выбранных функций интеграции. Сервис будет использовать токен только для функций, связанных с подключенной интеграцией. Вы можете удалить токен в настройках интеграции.
```

## 7. Хранение токенов

Токены и ключи доступа требуют отдельной защиты.

| Проверка | Статус | Комментарий |
|---|---|---|
| Токены не хранятся в открытом виде без необходимости | [ ] | Предпочтительно шифрование |
| Для токенов используется шифрование или безопасное хранилище | [ ] | Если проект позволяет |
| Ключ шифрования не хранится рядом с данными | [ ] | Не в публичном репозитории |
| Токены не попадают в логи | [ ] | Маскировать |
| Токены не попадают в email/уведомления | [ ] | Никогда не отправлять целиком |
| Токены не возвращаются в API целиком | [ ] | Только маска или статус подключения |
| Есть ограничение доступа сотрудников к токенам | [ ] | Роли и права |
| Есть дата создания/обновления токена | [ ] | Для аудита |
| Есть дата последней успешной проверки токена | [ ] | Полезно для диагностики |
| Есть статус токена | [ ] | active, invalid, revoked, expired |

Пример маскирования:

```text
abcd********wxyz
```

## 16. Хранение согласия на обработку токенов

При подключении интеграции рекомендуется фиксировать согласие пользователя.

Типовые документы:

```text
data_processing
api_terms
bank_data_processing
privacy
```

Рекомендуемые поля:

| Поле | Нужно | Комментарий |
|---|---|---|
| `user_id` | если есть | ID пользователя |
| `account_id` | если есть | ID аккаунта/tenant |
| `integration_id` | желательно | ID интеграции |
| `provider` | желательно | ozon, yandex_market, bank, crm и т.д. |
| `document_type` | да | data_processing, api_terms, bank_data_processing |
| `document_version` | да | Версия документа |
| `document_date` | желательно | Дата редакции |
| `checkbox_text` | желательно | Текст подтверждения |
| `accepted_at` | да | Дата принятия |
| `ip_address` | желательно | IP пользователя |
| `user_agent` | желательно | Браузер/клиент |
| `source` | да | settings, oauth_connection, marketplace_connection, bank_connection |
| `meta_json` | если нужно | scopes, provider, access_type |

Пример `meta_json`:

```json
{
  "provider": "ozon",
  "access_type": "api_key",
  "data_scope": ["orders", "products", "stocks", "messages"],
  "write_actions": ["update_stocks", "send_codes"]
}
```

## 17. Типовые ошибки

| Ошибка | Почему это плохо |
|---|---|
| Токен хранится в открытом виде без защиты | Риск компрометации внешнего аккаунта пользователя |
| Токен попадает в логи | Критичная утечка секрета |
| Токен передается через GET-параметр | Может попасть в логи, историю браузера и referer |
| Токен отображается целиком в интерфейсе | Риск утечки через скриншоты и доступ сотрудников |
| В документах не описано использование токена | Пользователь не понимает, что разрешает сервису |
| Нет возможности удалить токен | Пользователь не контролирует интеграцию |
| Не описаны действия сервиса от имени пользователя | Риск претензий при автоматических операциях |
| Не описаны сбои внешнего API | Непонятна ответственность при недоступности интеграции |
| Refresh token хранится как обычный токен | Повышенный риск безопасности |
| Webhook secret не используется | Риск поддельных событий |
| Секреты попали в репозиторий | Критичный инцидент безопасности |
