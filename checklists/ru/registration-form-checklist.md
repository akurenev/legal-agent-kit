

# Чек-лист формы регистрации

## Назначение

Этот чек-лист помогает проверить, что форма регистрации сайта, SaaS-сервиса, PWA, личного кабинета или бота корректно собирает согласия пользователя и содержит необходимые ссылки на правовые документы.

Чек-лист используется AI-агентом, разработчиком или владельцем проекта перед запуском регистрации или после изменения документов.

---

## 1. Когда использовать чек-лист

Используйте этот чек-лист, если в проекте есть:

- регистрация по email;
- регистрация по телефону;
- регистрация через Telegram/MAX;
- регистрация через OAuth;
- регистрация через приглашение;
- создание личного кабинета;
- создание аккаунта организации;
- регистрация сотрудника клиента;
- регистрация администратора клиента;
- регистрация через PWA или WebApp.

---

## 2. Документы, которые обычно нужны при регистрации

| Документ | Нужен | Комментарий |
|---|---|---|
| Пользовательское соглашение | [да / нет / уточнить] | Нужно, если пользователь создает аккаунт или получает доступ к сервису |
| Политика конфиденциальности | [да / нет / уточнить] | Нужна при обработке персональных данных |
| Согласие на обработку персональных данных | [да / нет / уточнить] | Рекомендуется отдельным документом или отдельным чекбоксом |
| Согласие на рекламные сообщения | [да / нет / уточнить] | Нужно только при рекламных или маркетинговых сообщениях |
| Правила подписки | [да / нет / уточнить] | Если регистрация сразу включает пробный период или подписку |
| Публичная оферта | [да / нет / уточнить] | Если регистрация совмещена с оплатой или подключением платного тарифа |
| Правила обработки API-токенов | [да / нет / уточнить] | Если при регистрации пользователь сразу подключает интеграцию |
| Правила обработки банковских данных | [да / нет / уточнить] | Если при регистрации сразу подключается банк |

## 3. Ссылки на форме регистрации

| Проверка | Статус | Комментарий |
|---|---|---|
| На форме есть ссылка на Пользовательское соглашение | [ ] | До отправки формы |
| На форме есть ссылка на Политику конфиденциальности | [ ] | До отправки формы |
| На форме есть ссылка на Согласие на обработку персональных данных | [ ] | Если используется отдельный документ |
| Ссылки открываются без авторизации | [ ] | Пользователь должен видеть документы до регистрации |
| Ссылки открываются в новой вкладке или модальном окне | [ ] | Чтобы пользователь не потерял введенные данные |
| Ссылки не ведут на 404 | [ ] | Проверить вручную |
| Ссылки доступны на мобильной версии | [ ] | Особенно важно для PWA и WebApp |
| Тексты ссылок понятны пользователю | [ ] | Не использовать только «тут» или «подробнее» |

Рекомендуемые URL:

```text
/legal/terms
/legal/privacy
/legal/personal-data-consent
/legal/marketing-consent
```

## 4. Обязательные чекбоксы

Для регистрации обычно рекомендуется использовать отдельные чекбоксы.

| Чекбокс | Обязательный | Статус | Комментарий |
|---|---|---|---|
| Я принимаю Пользовательское соглашение | да | [ ] | Нужно для использования сервиса |
| Я ознакомлен с Политикой конфиденциальности | да | [ ] | Можно объединить с текстом о согласии, но лучше оставить явно |
| Я даю согласие на обработку персональных данных | да | [ ] | Лучше отдельный чекбокс |
| Я согласен получать информационные и рекламные сообщения | нет | [ ] | Не делать обязательным без необходимости |

Рекомендуемый текст:

```text
[ ] Я принимаю Пользовательское соглашение
[ ] Я ознакомлен с Политикой конфиденциальности
[ ] Я даю согласие на обработку персональных данных
```

Если есть рекламные сообщения:

```text
[ ] Я согласен получать информационные и рекламные сообщения
```

---

## 5. Требования к чекбоксам

| Проверка | Статус | Комментарий |
|---|---|---|
| Чекбоксы не отмечены заранее | [ ] | Пользователь должен сам выразить согласие |
| Без обязательных чекбоксов регистрация невозможна | [ ] | Проверить frontend и backend |
| Рекламный чекбокс не является обязательным | [ ] | Если реклама не является частью услуги |
| Ошибка при незаполненном чекбоксе понятна пользователю | [ ] | Например: «Необходимо принять условия» |
| Текст чекбокса содержит ссылки на документы | [ ] | Ссылки должны быть кликабельны |
| Текст чекбокса сохраняется или может быть восстановлен | [ ] | Для журнала согласий |
| Версия документа фиксируется при регистрации | [ ] | Например: `terms: 1.0` |
| Дата редакции документа фиксируется при регистрации | [ ] | Например: `2026-06-01` |

---

## 6. Валидация на frontend

| Проверка | Статус | Комментарий |
|---|---|---|
| Frontend проверяет обязательные чекбоксы | [ ] | Для удобства пользователя |
| Ошибки отображаются рядом с формой | [ ] | Пользователь понимает, что исправить |
| Ошибки доступны на мобильной версии | [ ] | Не скрываются за клавиатурой или модальным окном |
| Кнопка регистрации не активна без обязательных согласий | [ ] | Допустимо, но backend все равно должен проверять |
| При обновлении формы введенные данные не теряются | [ ] | Особенно если документы открываются в текущей вкладке |
| Рекламное согласие можно не отмечать | [ ] | Регистрация должна проходить без него |

---

## 7. Валидация на backend

Frontend-проверки недостаточно. Backend должен также проверять обязательные согласия.

| Проверка | Статус | Комментарий |
|---|---|---|
| Backend проверяет принятие Пользовательского соглашения | [ ] | Нельзя полагаться только на frontend |
| Backend проверяет согласие на обработку персональных данных | [ ] | Если требуется для регистрации |
| Backend не требует рекламного согласия | [ ] | Если оно необязательное |
| Backend возвращает понятную ошибку | [ ] | Например: `terms_required` |
| Ошибки логируются без персональных данных сверх необходимости | [ ] | Не логировать пароли и лишние данные |
| API регистрации документирован | [ ] | Если есть публичный или внутренний API |

Примеры кодов ошибок:

```text
terms_required
privacy_required
personal_data_consent_required
invalid_consent_version
```

---

## 8. Какие согласия фиксировать

При регистрации обычно фиксируются:

```text
terms
privacy
personal_data_consent
```

Если пользователь согласился на рекламу:

```text
marketing_consent
```

Если регистрация связана с подключением интеграции:

```text
data_processing
api_terms
```

Если регистрация связана с подключением банка:

```text
bank_data_processing
```

---

## 9. Что хранить в журнале согласий

Рекомендуемые поля:

| Поле | Нужно | Комментарий |
|---|---|---|
| `user_id` | да | ID пользователя после создания аккаунта |
| `account_id` | если есть | ID аккаунта/tenant для B2B/SaaS |
| `organization_id` | если есть | ID организации клиента |
| `document_type` | да | `terms`, `privacy`, `personal_data_consent` |
| `document_title` | желательно | Название документа на момент согласия |
| `document_version` | да | Версия документа |
| `document_date` | желательно | Дата редакции документа |
| `document_url` | желательно | URL документа |
| `checkbox_text` | желательно | Текст чекбокса на момент согласия |
| `event_type` | да | Обычно `accepted` |
| `accepted_at` | да | Дата и время согласия |
| `ip_address` | желательно | IP-адрес пользователя |
| `user_agent` | желательно | Браузер/клиент пользователя |
| `source` | да | `registration` |
| `locale` | если есть | Язык интерфейса |
| `meta_json` | если нужно | Дополнительные данные |


## 11. Регистрация через Telegram/MAX-бота

Если регистрация проходит через бот, нужно проверить отдельные моменты.

| Проверка | Статус | Комментарий |
|---|---|---|
| Бот показывает ссылки на документы до регистрации | [ ] | Или до использования функций |
| Пользователь явно принимает условия | [ ] | Кнопка, команда или WebApp |
| Сохраняется Telegram ID / MAX ID | [ ] | В основной таблице пользователя |
| В согласии указывается источник `bot` или `registration` | [ ] | Для журнала согласий |
| Сохраняется текст кнопки или сообщения | [ ] | Например: «Принять условия» |
| Есть команда или способ открыть документы повторно | [ ] | Например: «Правовые документы» |
| Есть команда или способ обратиться в поддержку | [ ] | Если требуется |

Пример сообщения:

```text
Перед регистрацией ознакомьтесь с Пользовательским соглашением и Политикой конфиденциальности. Нажимая «Принять условия», вы подтверждаете согласие с документами и даете согласие на обработку персональных данных.
```

## 12. Регистрация организации или B2B-аккаунта

Если пользователь регистрирует организацию или аккаунт компании, нужно проверить:

| Проверка | Статус | Комментарий |
|---|---|---|
| Понятно, кто принимает условия | [ ] | Физлицо, представитель ИП/юрлица |
| Указано, что пользователь подтверждает полномочия | [ ] | В соглашении или форме |
| Собираются данные организации | [ ] | ИНН, название, адрес, контакты |
| Описана обработка данных организации | [ ] | В соглашении/политике |
| Есть роли сотрудников | [ ] | Администратор, сотрудник, владелец |
| Описана ответственность администратора аккаунта | [ ] | Приглашение сотрудников, доступ к данным |
| Согласия привязываются к user_id и account_id | [ ] | Для B2B/SaaS |

Пример текста:

```text
Регистрируя организацию, пользователь подтверждает, что имеет необходимые полномочия действовать от имени указанной организации.
```

## 13. Регистрация с пробным периодом

Если регистрация автоматически запускает пробный период:

| Проверка | Статус | Комментарий |
|---|---|---|
| Указано, что начинается пробный период | [ ] | До завершения регистрации |
| Указана длительность пробного периода | [ ] | Например: 7 или 14 дней |
| Указаны ограничения пробного периода | [ ] | Если есть |
| Указано, что будет после окончания | [ ] | Доступ ограничится или нужно оплатить |
| Указано, можно ли получить пробный период повторно | [ ] | Если применимо |
| Есть ссылка на Правила подписки | [ ] | Если пробный период связан с подпиской |
| Фиксируется принятие правил подписки | [ ] | Если пробный период является частью подписки |

---

## 14. Регистрация с оплатой

Если регистрация совмещена с оплатой или выбором тарифа:

| Проверка | Статус | Комментарий |
|---|---|---|
| Пользователь видит тариф до регистрации/оплаты | [ ] | Название, цена, период |
| Есть ссылка на Публичную оферту | [ ] | До оплаты |
| Есть ссылка на Правила оплаты и возврата | [ ] | До оплаты |
| Есть ссылка на Правила подписки | [ ] | Если доступ временный |
| Фиксируется принятие оферты | [ ] | Отдельно от регистрации |
| Фиксируется принятие правил подписки | [ ] | Если подписка есть |
| Пользователь понимает, когда начнется доступ | [ ] | После оплаты, после подтверждения и т.д. |

---

## 15. Типовые ошибки

| Ошибка | Почему это плохо |
|---|---|
| Нет ссылки на Политику конфиденциальности | Пользователь не видит правила обработки данных до регистрации |
| Все согласия объединены в одну фразу | Сложно доказать отдельное согласие на разные цели |
| Чекбоксы отмечены заранее | Пользователь не выражает активное согласие |
| Рекламное согласие обязательно | Риск, если реклама не нужна для оказания услуги |
| Backend не проверяет чекбоксы | Проверку можно обойти через API |
| Не хранится версия документа | Нельзя доказать, какую редакцию принял пользователь |
| Не хранится текст чекбокса | Сложно восстановить смысл согласия |
| Документы открываются только после регистрации | Пользователь не может ознакомиться до принятия |
| В документах остались плейсхолдеры | Документы не готовы к публикации |
| В форме одно название сервиса, в документах другое | Возникает противоречие |

---

## 16. Финальная проверка формы регистрации

| Проверка | Статус |
|---|---|
| Все нужные документы подготовлены | [ ] |
| Документы доступны без авторизации | [ ] |
| Ссылки на документы есть на форме | [ ] |
| Обязательные чекбоксы не отмечены заранее | [ ] |
| Регистрация невозможна без обязательных согласий | [ ] |
| Рекламное согласие необязательное | [ ] |
| Frontend показывает понятные ошибки | [ ] |
| Backend проверяет обязательные согласия | [ ] |
| Согласия сохраняются в журнале | [ ] |
| Сохраняется версия документа | [ ] |
| Сохраняется дата и время согласия | [ ] |
| Сохраняется источник `registration` | [ ] |
| Сохраняется IP и user-agent | [ ] |
| Документы не содержат плейсхолдеров | [ ] |
| Вопросы переданы владельцу проекта или юристу | [ ] |

---

## 17. Итоговый вывод агента

После проверки агент должен подготовить краткий итог:

```text
Форма регистрации проверена.

Готово:
- [список выполненных пунктов]

Требует доработки:
- [список проблем]

Вопросы:
- [список вопросов]

Риски:
- [список рисков]

Документы, согласия и интерфейс регистрации требуют проверки владельцем проекта и юристом перед публикацией.
```
