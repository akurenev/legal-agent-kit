

# Чек-лист персональных данных

## Назначение

Этот чек-лист помогает проверить, что проект корректно описывает сбор, хранение, использование, передачу и удаление персональных данных пользователей.

Чек-лист используется для сайтов, SaaS-сервисов, PWA, Telegram/MAX-ботов, API-сервисов, маркетплейс-интеграций, сервисов подписки и проектов с банковскими или платежными данными.

Чек-лист не заменяет юридическую проверку и должен использоваться как практический ориентир для владельца проекта, разработчика и AI-агента.

---

## 1. Когда использовать чек-лист

Используйте этот чек-лист, если проект собирает или обрабатывает хотя бы один из видов данных:

- ФИО;
- email;
- телефон;
- адрес;
- IP-адрес;
- user-agent;
- cookies;
- Telegram ID;
- MAX ID;
- username в мессенджере;
- данные организации;
- платежные данные;
- банковские данные;
- API-токены;
- данные из маркетплейсов;
- сообщения пользователей;
- файлы пользователей;
- логи действий;
- данные поддержки;
- любые другие данные, которые могут относиться к пользователю.

## 5. Минимизация данных

Проект должен собирать только те данные, которые действительно нужны.

| Проверка | Статус | Комментарий |
|---|---|---|
| Для каждого поля есть понятная цель | [ ] | Не собирать данные «на всякий случай» |
| Обязательные поля действительно необходимы | [ ] | Например, телефон не обязателен, если достаточно email |
| Не собираются лишние паспортные/финансовые данные | [ ] | Если это не нужно для услуги |
| Пользователь понимает, какие данные обязательны | [ ] | UI должен отличать обязательные поля |
| Пользователь понимает, какие данные необязательны | [ ] | Например, рекламные предпочтения |
| Данные можно обновить или исправить | [ ] | В личном кабинете или через поддержку |

## 18. Типовые ошибки

| Ошибка | Почему это плохо |
|---|---|
| В политике указаны не все данные | Документ не отражает фактическую обработку |
| Нет отдельного согласия на обработку данных | Сложнее подтвердить согласие пользователя |
| Нет целей обработки | Непонятно, зачем сервис собирает данные |
| Собираются лишние обязательные данные | Нарушается принцип минимизации |
| Не описаны третьи лица | Пользователь не знает, куда передаются данные |
| Не описаны cookies и аналитика | Риск по прозрачности обработки |
| Не описаны API-токены | Риск для интеграционных сервисов |
| Не описаны банковские данные | Критичный риск для финансовых сервисов |
| Нет порядка отзыва согласия | Пользователь не понимает, как управлять данными |
| Не хранится версия согласия | Нельзя доказать, какую редакцию принял пользователь |
| В логах сохраняются токены или пароли | Критичный риск безопасности |
