

# Как использовать Legal Agent Kit с ChatGPT

## Назначение

Этот документ описывает, как использовать **Legal Agent Kit** в обычном чате с ChatGPT или другим AI-ассистентом.

Такой сценарий подходит, если нужно:

- понять, какие правовые документы нужны проекту;
- подготовить черновики документов без прямого доступа агента к коду;
- сформировать тексты для сайта, лендинга, SaaS, PWA, бота или API-сервиса;
- подготовить задачу для разработчика или Codex;
- проверить уже существующие документы;
- обновить документы после изменения функциональности.

Материалы, подготовленные через ChatGPT, являются черновиками и требуют юридической проверки перед публикацией.

---

## 1. Чем работа в ChatGPT отличается от Codex

Codex обычно работает с файлами проекта и может анализировать структуру репозитория.

ChatGPT чаще работает с тем, что пользователь сам передал в чат:

- описание проекта;
- скриншоты;
- тексты тарифов;
- текущие документы;
- фрагменты интерфейса;
- маршруты API;
- описание платежей и интеграций;
- реквизиты владельца;
- список собираемых данных.

Поэтому при работе через ChatGPT особенно важно сначала собрать анкету проекта, а не сразу просить готовые документы.

---

## 2. Рекомендуемый порядок работы

1. Описать проект.
2. Попросить ChatGPT заполнить анкету проекта.
3. Проверить и дополнить анкету.
4. Попросить определить комплект документов.
5. Попросить подготовить документы по одному или комплектом.
6. Попросить подготовить рекомендации по внедрению в интерфейс.
7. Попросить подготовить задачу для разработчика или Codex.
8. Передать документы на юридическую проверку.

---

## 3. Что подготовить перед началом

Желательно заранее собрать:

| Данные | Пример |
|---|---|
| Название проекта | `[НАЗВАНИЕ СЕРВИСА]` |
| Домен | `https://example.com` |
| Владелец проекта | ИП, ООО, самозанятый, физическое лицо |
| Реквизиты | ИНН, ОГРН/ОГРНИП, адрес, email |
| Описание сервиса | Что делает сервис и для кого |
| Пользователи | Физлица, ИП, юрлица, сотрудники компаний |
| Модель | B2B, B2C или смешанная |
| Регистрация | Есть или нет |
| Личный кабинет | Есть или нет |
| Оплата | Есть или нет |
| Подписка | Есть или нет |
| Пробный период | Есть или нет |
| Платежный провайдер | Т-Банк, ЮKassa, CloudPayments, Stripe, другой |
| Персональные данные | Email, телефон, ФИО, IP, cookies и т.д. |
| Интеграции | Банки, маркетплейсы, мессенджеры, CRM, API |
| Аналитика | Яндекс.Метрика, Google Analytics, Top.Mail.Ru и т.д. |
| Рассылки | Сервисные, информационные, рекламные |
| Поддержка | Каналы и условия поддержки |

Если часть данных неизвестна, можно сразу указать: `оставить плейсхолдеры`.

## 4. Первый промпт: заполнить анкету проекта

```text
Мне нужно подготовить правовые документы для проекта.
Используй подход Legal Agent Kit.

Сначала не создавай документы.
Сначала заполни анкету проекта по структуре LEGAL_PROJECT_QUESTIONNAIRE.md.
Если данных не хватает — оставляй [УТОЧНИТЬ] и формируй список вопросов.

Описание проекта:
[ОПИСАНИЕ ПРОЕКТА]

Данные владельца:
[РЕКВИЗИТЫ ИЛИ УТОЧНИТЬ]

Оплата и тарифы:
[ОПИСАНИЕ ИЛИ УТОЧНИТЬ]

Какие данные собираются:
[СПИСОК ИЛИ УТОЧНИТЬ]

Интеграции:
[СПИСОК ИЛИ УТОЧНИТЬ]

Результат:
1. Заполненная анкета проекта.
2. Список неизвестных данных.
3. Список вопросов ко мне.
4. Предварительный список юридических рисков.
```

---

## 5. Второй промпт: определить комплект документов

После заполнения анкеты используйте следующий промпт:

```text
На основании заполненной анкеты определи, какие правовые документы нужны проекту.

Используй матрицу выбора документов:
- регистрация;
- личный кабинет;
- оплата;
- подписка;
- пробный период;
- cookies и аналитика;
- API;
- webhook;
- API-токены;
- банковские данные;
- мессенджеры;
- рассылки;
- поддержка.

Результат выдай таблицей:
- документ;
- нужен / не нужен / уточнить;
- почему нужен;
- какие данные нужны для подготовки.

Документы пока не создавай.
```

## 6. Третий промпт: подготовить документы

Когда комплект документов определен, можно попросить подготовить черновики.

```text
Теперь подготовь черновики правовых документов для проекта.

Требования:
1. Пиши документы в Markdown.
2. У каждого документа укажи дату редакции и версию.
3. Используй единые термины во всех документах.
4. Не выдумывай неизвестные данные — оставляй плейсхолдеры [УТОЧНИТЬ].
5. Документы должны быть понятными для обычного пользователя.
6. Документы должны быть адаптированы под фактическую работу проекта.
7. После документов выведи список плейсхолдеров и вопросов.
8. Отдельно выведи список юридических рисков.

Нужные документы:
[СПИСОК ДОКУМЕНТОВ]

Подготовленные документы являются черновиками и требуют юридической проверки перед публикацией.
```

---

## 7. Промпт для подготовки одного документа

Иногда лучше готовить документы по одному.

```text
Подготовь документ: [НАЗВАНИЕ ДОКУМЕНТА].

Используй данные проекта из анкеты выше.

Требования:
1. Формат Markdown.
2. Дата редакции: [ДАТА].
3. Версия: [ВЕРСИЯ].
4. Не выдумывай неизвестные данные.
5. Используй плейсхолдер [УТОЧНИТЬ], если данных нет.
6. Стиль — понятный, структурированный, без лишней юридической сложности.
7. В конце укажи, какие данные нужно уточнить.

Документ является черновиком и требует юридической проверки перед публикацией.
```

## 8. Промпт для внедрения документов в сайт или сервис

После подготовки документов нужно понять, где их разместить.

```text
Теперь подготовь рекомендации по внедрению правовых документов в интерфейс проекта.

Проверь сценарии:
1. Футер сайта.
2. Форма регистрации.
3. Форма обратной связи.
4. Страница тарифов.
5. Страница оплаты.
6. Личный кабинет.
7. Настройки интеграций.
8. Подключение банка.
9. Telegram/MAX-бот.
10. API-документация.
11. Хранение фактов согласия.

Результат:
- какие ссылки добавить;
- какие чекбоксы добавить;
- какие тексты использовать;
- какие согласия фиксировать;
- какие данные хранить в журнале согласий;
- что передать разработчику.
```

## 9. Промпт для подготовки задачи разработчику

Если документы подготовлены в чате, можно сформировать задачу для разработчика или Codex.

```text
Сформируй техническую задачу для разработчика на внедрение правовых документов в проект.

Нужно описать:
1. Какие документы добавить.
2. Куда положить Markdown-версии.
3. Какие публичные HTML-страницы создать.
4. Какие ссылки добавить в футер.
5. Какие чекбоксы добавить на регистрацию.
6. Какие ссылки добавить на страницу оплаты.
7. Как фиксировать принятие документов.
8. Какую таблицу или структуру хранения согласий добавить.
9. Что нельзя менять в рабочей логике.
10. Как проверить результат.

ТЗ должно быть понятным для Codex или разработчика.
```

## 10. Промпт для аудита существующих документов

Если документы уже есть, можно проверить их качество.

```text
Проведи аудит существующих правовых документов.

Я передам тексты документов или ссылки/файлы.

Проверь:
1. Какие документы есть.
2. Какие документы отсутствуют.
3. Есть ли дата редакции и версия.
4. Совпадает ли название сервиса.
5. Совпадают ли реквизиты владельца.
6. Описаны ли персональные данные.
7. Описаны ли цели обработки данных.
8. Описаны ли cookies и аналитика.
9. Описаны ли платежи, подписки и возвраты.
10. Описаны ли API-токены и внешние интеграции.
11. Описаны ли банковские данные, если они используются.
12. Есть ли отдельное согласие на обработку персональных данных.
13. Есть ли отдельное согласие на рекламные сообщения, если есть рассылки.
14. Есть ли противоречия между документами.

Результат:
- таблица проверки;
- список проблем;
- список отсутствующих документов;
- список вопросов;
- рекомендации по исправлению.
```

## 11. Промпт для обновления документов

Используется, если в проект добавили новую функцию.

```text
В проект добавлена новая функция:
[ОПИСАНИЕ ФУНКЦИИ]

Проверь, какие правовые документы нужно обновить.

Учитывай:
- меняется ли состав персональных данных;
- меняются ли цели обработки;
- появляется ли оплата или подписка;
- появляется ли новый платежный провайдер;
- появляется ли API, webhook или интеграция;
- появляются ли API-токены;
- появляются ли банковские данные;
- появляются ли рассылки;
- нужно ли повторное согласие пользователей.

Результат:
1. Список документов, которые нужно обновить.
2. Какие разделы нужно изменить.
3. Нужна ли новая версия документа.
4. Нужно ли повторное согласие пользователей.
5. Какие вопросы нужно проверить с юристом.
```

## 12. Как передавать данные в ChatGPT

Лучше передавать данные блоками.

### Хороший формат

```text
Проект:
[описание]

Пользователи:
[кто пользуется]

Регистрация:
[как устроена]

Оплата:
[как устроена]

Данные:
[что собирается]

Интеграции:
[какие есть]

Что нужно получить:
[анкета / комплект документов / аудит / ТЗ]
```

### Нежелательный формат

```text
Сделай мне все юридические документы для сайта.
```

Такой запрос почти всегда приведет к слишком общим документам и большому количеству неизвестных условий.

---

## 13. Что важно не забыть

При работе через ChatGPT нужно отдельно проверить:

- реальные реквизиты владельца;
- соответствие тарифов документам;
- условия возврата;
- платежного провайдера;
- фактический состав персональных данных;
- внешние сервисы и интеграции;
- cookies, аналитику и рекламные пиксели;
- необходимость отдельного рекламного согласия;
- необходимость хранения фактов согласия;
- необходимость повторного согласия при изменении документов.
