

# Как использовать Legal Agent Kit с Codex

## Назначение

Этот документ описывает, как использовать **Legal Agent Kit** с Codex для подготовки, аудита и внедрения правовых документов в проект.

Подход подходит для:

- нового сайта;
- лендинга;
- SaaS-сервиса;
- PWA-приложения;
- Telegram/MAX-бота;
- API-сервиса;
- маркетплейс-интеграции;
- сервиса с платежами, подписками или банковскими данными.

---

## 1. Основной сценарий работы

Рекомендуемый порядок:

1. Подключить Legal Agent Kit к проекту или передать Codex ссылку на репозиторий.
2. Дать Codex описание проекта.
3. Попросить Codex сначала заполнить анкету проекта.
4. Затем попросить определить нужный комплект документов.
5. После этого подготовить черновики документов.
6. Затем подготовить рекомендации по внедрению документов в интерфейс.
7. Отдельно проверить документы с юристом перед публикацией.

Важно: не просите Codex сразу «создать все документы». Сначала лучше заставить агента пройти этап анализа и заполнения анкеты.

---

## 2. Варианты подключения Legal Agent Kit

### 2.1. Использовать по ссылке на GitHub

Самый простой вариант — передать Codex ссылку на репозиторий:

```text
Используй Legal Agent Kit как методическую базу:
https://github.com/USERNAME/legal-agent-kit
```

Этот вариант подходит, если вы работаете в Codex и хотите использовать открытый репозиторий без копирования файлов в проект.

### 2.2. Скопировать в проект

Можно скопировать репозиторий или его часть в проект:

```text
/docs/legal-kit/
```

Тогда Codex будет работать с локальными инструкциями:

```text
В проекте есть /docs/legal-kit/.
Используй эти инструкции для подготовки правовых документов.
```

### 2.3. Подключить как Git submodule

Если Legal Agent Kit используется во многих проектах, можно подключить его как submodule:

```bash
git submodule add https://github.com/USERNAME/legal-agent-kit docs/legal-kit
```

Плюсы:

- одна актуальная методическая база для разных проектов;
- можно обновлять kit отдельно от основного проекта;
- удобно использовать в агентных сценариях.

Минусы:

- нужно понимать работу с Git submodule;
- при переносе проекта нужно не забывать инициализировать submodule.

## 3. Что подготовить перед запуском Codex

Перед задачей желательно подготовить:

- краткое описание проекта;
- ссылку на сайт или локальный интерфейс, если есть;
- README проекта;
- описание тарифов;
- описание платежного процесса;
- список собираемых данных;
- список внешних интеграций;
- информацию о владельце проекта;
- текущие документы, если они уже есть;
- структуру маршрутов или API-документацию;
- описание форм регистрации, оплаты и обратной связи.

Если каких-то данных нет, Codex должен оставить плейсхолдеры и список вопросов.

---

## 4. Первый промпт для Codex: анализ проекта

Используйте этот промпт в начале работы.

```text
Изучи проект и подготовь правовой анализ перед созданием документов.

Используй Legal Agent Kit как методическую базу:
[ССЫЛКА НА РЕПОЗИТОРИЙ ИЛИ ПУТЬ /docs/legal-kit/]

Задача на первом этапе:
1. Изучи README.md, AGENTS.md, .env.example, маршруты, страницы регистрации, оплаты, личного кабинета, футер сайта и существующие правовые документы, если они есть.
2. Определи тип проекта: лендинг, сайт, SaaS, PWA, бот, API-сервис, маркетплейс-интеграция, сервис подписки, сервис с банковскими данными или другое.
3. Определи модель работы: B2B, B2C или смешанная.
4. Заполни анкету проекта по структуре LEGAL_PROJECT_QUESTIONNAIRE.md.
5. Определи, какие документы нужны, используя docs/document-matrix.md.
6. Не создавай документы на этом этапе.
7. Не меняй рабочий код проекта.
8. Не выдумывай неизвестные данные. Используй плейсхолдеры [УТОЧНИТЬ].
9. Подготовь список вопросов к владельцу проекта и список юридических рисков.

Результат:
- заполненная анкета проекта;
- список нужных документов с обоснованием;
- список неизвестных данных;
- список вопросов;
- список юридических рисков;
- рекомендации по следующему шагу.
```

## 5. Второй промпт: создание документов

После анализа и уточнения данных можно дать второй промпт.

```text
На основании выполненного анализа подготовь комплект правовых документов.

Используй:
- LEGAL_AGENT_GUIDE.md;
- docs/document-matrix.md;
- docs/legal-disclaimer.md;
- templates/ru/;
- заполненную анкету проекта.

Требования:
1. Создай документы в Markdown.
2. Готовые документы положи в /docs/legal/generated/.
3. Используй единые термины во всех документах.
4. У каждого документа укажи дату редакции и версию.
5. Не выдумывай неизвестные данные. Оставляй плейсхолдеры.
6. Для каждого документа добавь понятную структуру разделов.
7. Подготовь отдельный файл legal_questions.md со списком вопросов.
8. Подготовь отдельный файл legal_risks.md со списком юридических рисков.
9. Подготовь legal_integration_checklist.md с рекомендациями по внедрению документов в интерфейс.
10. Не меняй рабочий код проекта без отдельной команды.

Ожидаемые файлы:
- terms.md
- privacy.md
- personal_data_consent.md
- offer.md, если есть оплата или подписка
- payment_and_refund.md, если есть оплата
- subscription_rules.md, если есть подписка или пробный период
- cookie_policy.md, если есть cookies, аналитика или рекламные пиксели
- support_rules.md, если есть поддержка
- data_processing.md, если есть API-токены или внешние интеграции
- bank_data_processing.md, если есть банковские данные
- marketing_consent.md, если есть рекламные сообщения
- legal_questions.md
- legal_risks.md
- legal_integration_checklist.md

В финальном ответе укажи, какие документы созданы и какие данные требуют уточнения.
Обязательно укажи, что документы являются черновиками и требуют юридической проверки перед публикацией.
```

## 6. Третий промпт: внедрение документов в проект

После подготовки документов можно попросить Codex добавить страницы и ссылки.

```text
В проекте уже подготовлены правовые документы в /docs/legal/generated/.

Нужно подготовить план внедрения документов в интерфейс и, если возможно, внести безопасные изменения.

Используй:
- docs/interface-checklist.md;
- docs/consent-storage.md;
- подготовленные документы из /docs/legal/generated/.

Требования:
1. Найди, где в проекте находится футер сайта.
2. Найди страницы регистрации, оплаты, тарифов, личного кабинета и настроек интеграций.
3. Определи, какие ссылки и чекбоксы нужно добавить.
4. Сначала предложи план изменений без правки кода.
5. Не меняй бизнес-логику.
6. Не меняй существующие маршруты без необходимости.
7. Не ломай дизайн и текущие механизмы.
8. Если есть SPA/PWA — предложи компоненты или маршруты для правовых страниц.
9. Если есть backend-rendered HTML — предложи HTML-страницы.
10. Если есть регистрация или оплата — предложи механизм хранения согласий.

Результат:
- список файлов, которые нужно изменить;
- список ссылок, которые нужно добавить;
- тексты чекбоксов;
- рекомендации по хранению согласий;
- список рисков;
- только после отдельного подтверждения можно вносить изменения в код.
```

## 7. Промпт для аудита существующих документов

Если документы уже есть, используйте отдельную задачу на аудит.

```text
Проведи аудит существующих правовых документов проекта.

Используй Legal Agent Kit:
[ССЫЛКА ИЛИ ПУТЬ]

Проверь:
1. Какие документы уже есть.
2. Какие документы отсутствуют.
3. Есть ли дата редакции и версия.
4. Совпадают ли реквизиты владельца во всех документах.
5. Совпадает ли название сервиса.
6. Описаны ли персональные данные и цели обработки.
7. Описаны ли платежи, подписки, возвраты и пробный период.
8. Описаны ли cookies, аналитика и рекламные системы.
9. Описаны ли API-токены, банковские данные и внешние интеграции.
10. Есть ли ссылки на документы в интерфейсе.
11. Есть ли чекбоксы согласий.
12. Фиксируются ли согласия пользователей.

Не меняй документы и код на первом этапе.

Результат:
- таблица найденных документов;
- список отсутствующих документов;
- список противоречий;
- список юридических рисков;
- список рекомендаций;
- план исправлений.
```

## 8. Промпт для обновления документов после изменения сервиса

Используйте, если в проект добавили оплату, подписку, банк, API, маркетплейс, рассылки или новый тип данных.

```text
В проекте изменилась функциональность. Нужно проверить, какие правовые документы нужно обновить.

Новое изменение:
[ОПИСАНИЕ ИЗМЕНЕНИЯ]

Используй Legal Agent Kit:
[ССЫЛКА ИЛИ ПУТЬ]

Порядок работы:
1. Определи, какие документы затрагивает изменение.
2. Проверь, нужно ли обновить анкету проекта.
3. Проверь, нужно ли изменить Политику конфиденциальности.
4. Проверь, нужно ли обновить Пользовательское соглашение.
5. Проверь, нужно ли обновить Оферту, Правила оплаты, Правила подписки.
6. Проверь, нужно ли добавить правила обработки токенов, банковских данных или данных интеграций.
7. Предложи новую версию документов.
8. Укажи, нужно ли повторное согласие пользователей.
9. Не меняй документы и код без отдельного подтверждения.

Результат:
- список затронутых документов;
- рекомендуемые изменения;
- новая версия документов;
- нужно ли повторное согласие;
- список вопросов к владельцу проекта или юристу.
```

## 9. Рекомендуемые коммиты

Если Codex вносит изменения в репозиторий, лучше разделять работу на несколько коммитов.

Примеры:

```bash
git add docs/legal/generated
git commit -m "Add legal document drafts"
```

```bash
git add public/legal
 git commit -m "Add public legal pages"
```

```bash
git add src components pages
 git commit -m "Add legal links and consent checkboxes"
```

```bash
git add database migrations
 git commit -m "Add user consent tracking"
```

Не стоит смешивать в одном коммите:

- тексты документов;
- изменения интерфейса;
- миграции базы;
- бизнес-логику оплаты;
- рефакторинг.

---

## 10. Что Codex не должен делать без подтверждения

Codex не должен без отдельного подтверждения:

- публиковать документы на production;
- менять платежную логику;
- менять регистрацию;
- менять обработку персональных данных;
- удалять существующие документы;
- заменять реальные реквизиты на неизвестные;
- подключать новые рассылки;
- включать обязательное рекламное согласие;
- менять маршруты API;
- менять базу данных;
- делать массовый рефакторинг.

---

## 11. Минимальный рабочий сценарий

Если нужно быстро подготовить документы для нового проекта, используйте короткий сценарий:

```text
Используй Legal Agent Kit.

Сначала заполни LEGAL_PROJECT_QUESTIONNAIRE.md по моему проекту.
Потом определи комплект документов по docs/document-matrix.md.
Потом подготовь документы в /docs/legal/generated/.
Если данных не хватает — оставь плейсхолдеры и список вопросов.
Не меняй рабочий код.
```

---

## 12. Финальная проверка результата Codex

После работы Codex проверьте:

- создана ли анкета проекта;
- есть ли список нужных документов;
- созданы ли документы в правильной папке;
- нет ли выдуманных реквизитов;
- есть ли плейсхолдеры для неизвестных данных;
- есть ли список вопросов;
- есть ли список юридических рисков;
- есть ли рекомендации по внедрению;
- есть ли рекомендации по хранению согласий;
- указано ли, что документы требуют юридической проверки.

---

## 13. Обязательная финальная пометка

Любой результат Codex по этому kit должен содержать пометку:

```text
Подготовленные документы являются черновиками и требуют юридической проверки перед публикацией.
```
