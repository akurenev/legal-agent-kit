

# Чек-лист внедрения правовых документов в интерфейс

## Назначение

Этот чек-лист помогает проверить, что подготовленные правовые документы не просто лежат в проекте, а корректно встроены в сайт, сервис, PWA, бот, личный кабинет, страницу оплаты и API-документацию.

AI-агент должен использовать этот файл после подготовки документов, чтобы сформировать рекомендации для разработчика или проверить уже существующее внедрение.

---

## 1. Общие требования

| Проверка | Статус | Комментарий |
|---|---|---|
| Все нужные документы созданы | [ ] | Проверить по `docs/document-matrix.md` |
| У каждого документа есть дата редакции | [ ] | Например: `Редакция от: [ДАТА]` |
| У каждого документа есть версия | [ ] | Например: `Версия: 1.0` |
| Во всех документах одинаковое название сервиса | [ ] | Не должно быть разных вариантов названия |
| Во всех документах одинаковые реквизиты владельца | [ ] | ИНН, ОГРН/ОГРНИП, адрес, email |
| Все неизвестные данные оставлены как плейсхолдеры | [ ] | Агент не должен выдумывать данные |
| Сформирован список плейсхолдеров | [ ] | Отдельный файл или раздел в ответе |
| Сформирован список вопросов к владельцу проекта | [ ] | Например: `legal_questions.md` |
| Сформирован список юридических рисков | [ ] | Например: `legal_risks.md` |
| Документы доступны пользователю до регистрации и оплаты | [ ] | Ссылки должны открываться без авторизации |

## 2. Футер сайта

Если у проекта есть публичный сайт, лендинг или веб-интерфейс, в футере рекомендуется разместить ссылки на актуальные документы.

| Ссылка | Нужна | Статус | Комментарий |
|---|---|---|---|
| Пользовательское соглашение | [да / нет / уточнить] | [ ] | Обычно нужно при регистрации или личном кабинете |
| Политика конфиденциальности | [да / нет / уточнить] | [ ] | Нужна при сборе персональных данных |
| Согласие на обработку персональных данных | [да / нет / уточнить] | [ ] | Нужно при формах, регистрации, заявках |
| Публичная оферта | [да / нет / уточнить] | [ ] | Нужна при продаже услуг или подписке |
| Правила оплаты и возврата | [да / нет / уточнить] | [ ] | Нужны при онлайн-оплате |
| Правила подписки | [да / нет / уточнить] | [ ] | Нужны при подписке или пробном периоде |
| Политика cookies | [да / нет / уточнить] | [ ] | Нужна при cookies, аналитике, пикселях |
| Правила поддержки | [да / нет / уточнить] | [ ] | Нужны при заявленной поддержке или SLA |
| Контакты | [да / нет / уточнить] | [ ] | Email, телефон или форма обращения |

Рекомендации:

- ссылки должны быть видны на всех публичных страницах;
- ссылки должны открываться без авторизации;
- URL документов должны быть стабильными;
- при изменении документа нужно обновлять дату редакции и версию;
- старые редакции желательно сохранять, если сервис фиксирует согласия по версиям.

## 3. Страница регистрации

Если в проекте есть регистрация, нужно проверить юридически значимые элементы формы.

### 3.1. Обязательные ссылки и чекбоксы

| Элемент | Статус | Комментарий |
|---|---|---|
| Есть ссылка на Пользовательское соглашение | [ ] | Текст ссылки должен быть понятным |
| Есть ссылка на Политику конфиденциальности | [ ] | Пользователь должен видеть ее до регистрации |
| Есть отдельное Согласие на обработку персональных данных | [ ] | Не смешивать с пользовательским соглашением |
| Чекбоксы не проставлены заранее | [ ] | Пользователь должен сам выразить согласие |
| Без обязательных согласий регистрация невозможна | [ ] | Для документов, необходимых для оказания услуги |
| Текст чекбоксов сохранен или может быть восстановлен | [ ] | Нужно для доказательства согласия |
| Фиксируется версия документа | [ ] | Например: `terms: 1.0` |
| Фиксируется дата и время согласия | [ ] | Желательно с таймзоной |
| Фиксируется IP-адрес | [ ] | Для журнала согласий |
| Фиксируется user-agent | [ ] | Для журнала согласий |

### 3.2. Рекомендуемые тексты

```text
[ ] Я принимаю Пользовательское соглашение
[ ] Я ознакомлен с Политикой конфиденциальности
[ ] Я даю согласие на обработку персональных данных
```

Если есть рекламные или информационные рассылки:

```text
[ ] Я согласен получать информационные и рекламные сообщения
```

Важно: согласие на рекламные сообщения не должно быть обязательным, если рассылка не является необходимой частью услуги.

## 4. Форма обратной связи или заявки

Если сайт собирает заявки без регистрации, нужно проверить форму.

| Проверка | Статус | Комментарий |
|---|---|---|
| Под формой есть ссылка на Политику конфиденциальности | [ ] | Обязательно при сборе персональных данных |
| Есть согласие на обработку персональных данных | [ ] | Чекбокс или явный текст согласия |
| Понятно, какие данные отправляются | [ ] | Имя, телефон, email, комментарий |
| Указана цель обработки | [ ] | Например: обработка заявки и обратная связь |
| Если есть рекламная рассылка — согласие отдельное | [ ] | Не смешивать с обработкой заявки |
| Форма не отправляется без обязательного согласия | [ ] | Если согласие требуется |

Пример текста:

```text
Нажимая «Отправить», я даю согласие на обработку персональных данных и соглашаюсь с Политикой конфиденциальности.
```

Для более строгого сценария лучше использовать отдельный чекбокс:

```text
[ ] Я даю согласие на обработку персональных данных для обработки моего обращения
```

## 5. Страница оплаты

Если есть онлайн-оплата, подписка, тарифы или пробный период, страница оплаты должна содержать ссылки на платежные документы.

| Проверка | Статус | Комментарий |
|---|---|---|
| Есть ссылка на Публичную оферту | [ ] | До оплаты |
| Есть ссылка на Правила оплаты и возврата | [ ] | До оплаты |
| Есть ссылка на Правила подписки | [ ] | Если доступ предоставляется на период |
| Пользователь видит название тарифа | [ ] | Должно совпадать с офертой |
| Пользователь видит стоимость | [ ] | Цена, валюта, период |
| Пользователь видит период доступа | [ ] | Например: 30 дней, месяц, год |
| Пользователь видит условия продления | [ ] | Особенно при автопродлении |
| Пользователь видит условия возврата | [ ] | Или ссылку на правила возврата |
| Понятно, когда услуга считается оказанной | [ ] | Важно для возвратов |
| Фиксируется принятие оферты | [ ] | Версия, дата, IP, user-agent |
| Платежный провайдер указан корректно | [ ] | В оферте или правилах оплаты |
| Указано, как пользователь получает чек | [ ] | Если применимо |

Рекомендуемый текст перед оплатой:

```text
Нажимая «Оплатить», я принимаю условия Публичной оферты и Правила оплаты и возврата.
```

Если есть подписка:

```text
Нажимая «Оплатить», я принимаю условия Публичной оферты, Правила подписки и Правила оплаты и возврата.
```

Если есть автопродление, нужно явно указать:

```text
Подписка продлевается автоматически до ее отключения пользователем. Условия автопродления описаны в Правилах подписки.
```

## 6. Страница тарифов

Если на сайте есть тарифы, агент должен проверить соответствие тарифов документам.

| Проверка | Статус | Комментарий |
|---|---|---|
| Названия тарифов совпадают с офертой | [ ] | Не должно быть разных названий |
| Стоимость совпадает с офертой | [ ] | Если цены указаны в оферте |
| Период подписки указан явно | [ ] | День, месяц, год, количество дней |
| Ограничения тарифа описаны понятно | [ ] | Лимиты, функции, пользователи, объемы |
| Пробный период описан отдельно | [ ] | Срок, ограничения, окончание доступа |
| Бесплатный тариф описан отдельно | [ ] | Если есть |
| Условия перехода между тарифами описаны | [ ] | Если применимо |
| Условия отключения подписки доступны | [ ] | Ссылка на правила подписки |
| Нет маркетинговых обещаний, которых нет в документах | [ ] | Например, «гарантированный результат» |

## 7. Личный кабинет

Если в проекте есть личный кабинет, рекомендуется добавить раздел с правовыми документами и историей согласий.

| Проверка | Статус | Комментарий |
|---|---|---|
| Есть раздел «Правовые документы» | [ ] | В настройках или профиле |
| Пользователь видит актуальные документы | [ ] | Ссылки на текущие версии |
| Пользователь видит дату редакции документов | [ ] | Желательно |
| Пользователь может скачать документы | [ ] | Не обязательно, но полезно |
| Пользователь видит принятые документы | [ ] | Если хранится история согласий |
| Пользователь может запросить удаление аккаунта | [ ] | Если предусмотрено проектом |
| Пользователь может отозвать необязательные согласия | [ ] | Например, рекламные сообщения |
| Пользователь может отключить интеграции | [ ] | Если есть внешние подключения |
| Пользователь может удалить API-токены | [ ] | Если сервис хранит токены |
| Пользователь может отключить банк | [ ] | Если сервис подключается к банку |

## 8. Настройки API и интеграций

Если проект работает с внешними сервисами, токенами, API-ключами, webhook или OAuth, в интерфейсе должны быть понятные предупреждения.

| Проверка | Статус | Комментарий |
|---|---|---|
| Пользователь понимает, какой внешний сервис подключает | [ ] | Например: банк, маркетплейс, CRM |
| Пользователь понимает, какие данные будут доступны сервису | [ ] | До подключения |
| Есть ссылка на документ об обработке токенов/данных интеграций | [ ] | Если используется |
| Указано, для чего используются токены | [ ] | Например: загрузка заказов, отправка событий |
| Указано, что пользователь отвечает за корректность токенов | [ ] | Если применимо |
| Есть возможность удалить токен | [ ] | Рекомендуется |
| Есть возможность отключить интеграцию | [ ] | Рекомендуется |
| Есть предупреждение о сбоях внешних API | [ ] | Сервис не всегда контролирует внешнюю сторону |
| Для webhook указано, кто отвечает за принимающий URL | [ ] | Клиент отвечает за доступность своего endpoint |

Пример текста для API-токена:

```text
Передавая API-токен, вы разрешаете сервису использовать его для выполнения выбранных функций интеграции. Вы можете удалить токен в настройках интеграции.
```

## 9. Подключение банка или финансовых данных

Если сервис получает банковские данные, интерфейс подключения должен быть особенно понятным.

| Проверка | Статус | Комментарий |
|---|---|---|
| Пользователь видит название подключаемого банка | [ ] | До авторизации |
| Пользователь видит, какие данные получит сервис | [ ] | Выписки, операции, остатки, контрагенты |
| Указано, может ли сервис инициировать платежи | [ ] | Если только чтение — указать явно |
| Есть ссылка на правила обработки банковских данных | [ ] | До подключения |
| Есть возможность отключить банк | [ ] | В личном кабинете |
| Есть возможность удалить или запросить удаление данных | [ ] | Если предусмотрено |
| Указан срок хранения банковских данных | [ ] | В документе или интерфейсе |
| Указано, что сервис не является финансовым/налоговым консультантом | [ ] | Если сервис дает аналитику или рекомендации |
| Указаны ограничения ответственности за решения пользователя | [ ] | Если пользователь принимает решения на основе данных |

Пример текста:

```text
Сервис получает данные по счетам только для отображения уведомлений, сводок и аналитики. Сервис не инициирует платежи и не распоряжается денежными средствами, если иное явно не указано в условиях сервиса.
```

## 10. Бот Telegram / MAX

Если сервис работает через бота, нужно проверить документы и сообщения внутри бота.

| Проверка | Статус | Комментарий |
|---|---|---|
| При первом запуске бот показывает ссылки на документы | [ ] | Или до начала использования функций |
| Пользователь принимает условия использования | [ ] | Команда, кнопка или чекбокс в WebApp |
| Есть ссылка на Политику конфиденциальности | [ ] | До передачи данных |
| Есть согласие на обработку персональных данных | [ ] | Если собираются данные |
| Указано, какие идентификаторы мессенджера сохраняются | [ ] | Telegram ID, MAX ID, username |
| Указано, какие уведомления отправляет бот | [ ] | Сервисные, информационные, рекламные |
| Есть возможность отключить уведомления | [ ] | Если применимо |
| Есть команда или инструкция для связи с поддержкой | [ ] | Например: `/support` |
| Если бот принимает оплату — есть ссылка на оферту | [ ] | До оплаты |

Пример сообщения при первом запуске:

```text
Перед использованием сервиса ознакомьтесь с Пользовательским соглашением и Политикой конфиденциальности. Продолжая работу с ботом, вы подтверждаете согласие с условиями сервиса.
```

Для более строгого сценария лучше использовать кнопку:

```text
[Принять условия]
```

## 12. Хранение факта согласия

Для проектов с регистрацией, оплатой, подпиской, ботом или личным кабинетом рекомендуется хранить историю согласий.

### 12.1. Рекомендуемые поля

| Поле | Описание |
|---|---|
| `id` | ID записи |
| `user_id` | ID пользователя |
| `document_type` | Тип документа |
| `document_version` | Версия документа |
| `document_date` | Дата редакции документа |
| `document_url` | URL документа |
| `checkbox_text` | Текст чекбокса или подтверждения |
| `accepted_at` | Дата и время принятия |
| `ip_address` | IP-адрес пользователя |
| `user_agent` | User-agent пользователя |
| `source` | registration, payment, profile, bot, api |

### 12.2. Типы документов

```text
terms
privacy
personal_data_consent
offer
payment_and_refund
subscription_rules
marketing_consent
cookie_policy
data_processing
bank_data_processing
```

### 12.3. Что важно проверить

| Проверка | Статус | Комментарий |
|---|---|---|
| Согласия сохраняются в базе | [ ] | Или в другом надежном журнале |
| Сохраняется версия документа | [ ] | Важно при обновлениях |
| Сохраняется дата редакции документа | [ ] | Важно при спорах |
| Сохраняется текст чекбокса | [ ] | Позволяет восстановить смысл согласия |
| Сохраняется источник согласия | [ ] | Регистрация, оплата, бот и т.д. |
| Сохраняется IP и user-agent | [ ] | Желательно |
| Есть механизм повторного согласия | [ ] | При существенных изменениях документов |

## 13. Обновление документов

Если документы изменяются, нужно проверить процесс обновления.

| Проверка | Статус | Комментарий |
|---|---|---|
| У документа обновлена версия | [ ] | Например: 1.0 → 1.1 |
| У документа обновлена дата редакции | [ ] | Дата публикации новой редакции |
| Старая редакция сохранена | [ ] | Желательно для истории |
| Пользователи уведомлены об изменениях | [ ] | Если изменения существенные |
| Запрошено повторное согласие | [ ] | Если изменения требуют нового согласия |
| В интерфейсе обновлены ссылки | [ ] | Если URL изменился |
| В базе фиксируется новая версия согласия | [ ] | Если пользователь принял новую редакцию |

---

## 14. Финальная проверка внедрения

Перед публикацией проекта агент или разработчик должен проверить:

- все необходимые документы созданы;
- все документы доступны публично;
- ссылки открываются без авторизации;
- ссылки добавлены в футер;
- формы регистрации и заявки содержат нужные согласия;
- страница оплаты содержит ссылки на оферту и правила оплаты;
- страница тарифов не противоречит оферте;
- личный кабинет содержит раздел с документами или ссылками;
- интеграции содержат предупреждения о токенах и внешних сервисах;
- подключение банка содержит понятное описание доступа к данным;
- бот показывает ссылки на документы до использования сервиса;
- API-документация содержит условия использования;
- согласия пользователей фиксируются;
- версии документов хранятся;
- подготовлен список вопросов для юриста.

## 15. Итоговый вывод агента

После проверки агент должен подготовить краткий итог:

```text
Проверено внедрение правовых документов.

Нужно добавить:
- [список недостающих ссылок, чекбоксов или страниц]

Нужно уточнить:
- [список вопросов]

Юридические риски:
- [список рисков]

Документы и внедрение требуют проверки владельцем проекта и юристом перед публикацией.
```
