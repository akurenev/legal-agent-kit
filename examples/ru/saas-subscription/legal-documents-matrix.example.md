

# Пример матрицы правовых документов: SaaS с подпиской

## Назначение примера

Этот пример показывает, как AI-агент может определить комплект правовых документов для SaaS-сервиса с регистрацией, личным кабинетом, оплатой, подпиской, пробным периодом, API, webhook, уведомлениями и внешними интеграциями.

Пример основан на демонстрационной анкете проекта:

```text
examples/ru/saas-subscription/project-questionnaire.example.md
```

Все данные ниже демонстрационные. Итоговый комплект документов для реального проекта должен быть проверен владельцем проекта и юристом.

## 2. Итоговая матрица документов

| Документ | Нужен | Причина | Какие данные нужны |
|---|---|---|---|
| Пользовательское соглашение | да | Есть регистрация, личный кабинет, роли пользователей, API, интеграции и ограничения использования | Название сервиса, владелец, реквизиты, роли, правила блокировки, порядок удаления аккаунта |
| Политика конфиденциальности | да | Сервис обрабатывает email, ФИО, IP, user-agent, cookies, данные аккаунта, платежные данные, API-токены и данные интеграций | Реквизиты оператора, состав данных, цели обработки, сроки хранения, третьи лица, аналитика, провайдеры |
| Согласие на обработку персональных данных | да | Есть регистрация, формы, поддержка, уведомления и оплата | Перечень данных, цели обработки, срок действия согласия, порядок отзыва |
| Публичная оферта | да | Есть платные тарифы и подписка | Исполнитель, тарифы, стоимость, период доступа, порядок акцепта, момент оказания услуги, возвраты |
| Правила оплаты и возврата | да | Есть онлайн-оплата, платежный провайдер, возможные возвраты и чеки | Платежный провайдер, фискализация, чек, условия возврата, срок рассмотрения обращения |
| Правила подписки | да | Есть подписка на 30 дней, пробный период и возможный льготный период | Период подписки, пробный период, льготный период, продление, окончание доступа, смена тарифа |
| Политика cookies | да | Используются cookies и аналитика | Список cookies, аналитика, пиксели, cookie-баннер, сроки хранения, отключение cookies |
| Правила поддержки | да | Есть поддержка пользователей по сервису, оплате, интеграциям и API | Каналы поддержки, время работы, сроки реакции, SLA, что входит и не входит в поддержку |
| Согласие на информационные и рекламные сообщения | уточнить | Нужно, если сервис отправляет рекламные или маркетинговые сообщения | Каналы рассылки, виды сообщений, способ отписки, факт согласия и отзыва |
| Правила обработки API-токенов и данных интеграций | да | Есть API, webhook и внешние интеграции, пользователь может передавать токены | Типы токенов, цели использования, хранение, удаление, внешние сервисы, ответственность пользователя |
| Правила обработки банковских данных | нет | В базовом примере сервис не получает банковские данные | Если банк появится позже — банк, OAuth/API, состав данных, read-only или платежные действия |
| Условия использования API | да | Сервис предоставляет API внешним клиентам | Авторизация, лимиты, методы, webhook, логи, изменения API, блокировка доступа |
| Правила использования бота | да | Есть Telegram-бот для уведомлений и быстрых команд | Бот, каналы, команды, уведомления, данные мессенджера, отписка, поддержка через бота |

## 3. Документы, которые нужно подготовить в первую очередь

Для запуска такого SaaS-сервиса минимальный рабочий комплект:

```text
terms.md
privacy.md
personal_data_consent.md
offer.md
payment_and_refund.md
subscription_rules.md
cookie_policy.md
support_rules.md
data_processing.md
api_terms.md
bot_terms.md
```

Дополнительно, если будут рекламные сообщения:

```text
marketing_consent.md
```

Документ для банковских данных пока не нужен:

```text
bank_data_processing.md — не требуется, пока сервис не получает банковские данные.
```

## 6. Рекомендуемые тексты чекбоксов

### Регистрация

```text
[ ] Я принимаю Пользовательское соглашение
[ ] Я ознакомлен с Политикой конфиденциальности
[ ] Я даю согласие на обработку персональных данных
```

### Рекламные сообщения

```text
[ ] Я согласен получать информационные и рекламные сообщения
```

Этот чекбокс должен быть необязательным, если рекламные сообщения не являются необходимой частью услуги.

### Оплата подписки

```text
Нажимая «Оплатить», я принимаю условия Публичной оферты, Правила подписки и Правила оплаты и возврата.
```

### Интеграция

```text
Передавая API-токен, вы разрешаете сервису использовать его для выполнения выбранных функций интеграции. Вы можете удалить токен в настройках интеграции.
```

### Webhook

```text
Пользователь самостоятельно отвечает за корректность и доступность указанного webhook URL.
```

| Подключение trial | terms, subscription_rules | registration / profile |
| Подключение API-интеграции | data_processing, api_terms | settings / oauth_connection |
| Настройка webhook | api_terms, data_processing | settings |
| Подключение Telegram-бота | bot_terms, privacy, personal_data_consent | bot |
| Рекламное согласие | marketing_consent | registration / profile / subscription_form |
| Отзыв рекламного согласия | marketing_consent, event_type=revoked | profile / unsubscribe / bot |

## 10. Риски для юридической проверки

| Риск | Почему важно |
|---|---|
| Не определен момент оказания услуги | Влияет на возвраты и оферту |
| Не определены условия возврата | Риск споров по оплате |
| Не ясно, есть ли автопродление | Существенное условие для пользователя |
| Тарифы могут отличаться в интерфейсе и оферте | Риск противоречий |
| Нет отдельного рекламного согласия при рекламных рассылках | Риск по рассылкам |
| Не описаны API-токены и webhook | Риск по интеграциям и безопасности |
| Не описаны сроки хранения данных | Риск по политике конфиденциальности |
| Не фиксируются версии принятых документов | Сложно доказать согласие пользователя |
| Не ясно, кто отвечает за webhook endpoint клиента | Риск претензий при недоставке событий |
| Не описана зависимость от внешних API | Риск претензий при сбоях интеграций |

## 11. Итоговый вывод

Для Demo SaaS требуется полный комплект документов для SaaS с подпиской:

```text
terms.md
privacy.md
personal_data_consent.md
offer.md
payment_and_refund.md
subscription_rules.md
cookie_policy.md
support_rules.md
data_processing.md
api_terms.md
bot_terms.md
```

Документ `marketing_consent.md` нужен, если сервис отправляет рекламные или маркетинговые сообщения.

Документ `bank_data_processing.md` в текущем примере не нужен, потому что банковские данные не обрабатываются. Если в сервис будет добавлена банковская интеграция, матрицу нужно обновить.

Перед публикацией документов нужно уточнить реквизиты владельца, платежного провайдера, тарифы, возвраты, автопродление, сроки хранения данных, аналитику и технические параметры API.

Документы являются черновиками и требуют юридической проверки перед публикацией.
