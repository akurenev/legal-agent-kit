

# Пример чек-листа внедрения документов: простой лендинг

## Назначение примера

Этот пример показывает, как AI-агент может подготовить чек-лист внедрения правовых документов в простой лендинг, сайт-визитку или промо-страницу.

Пример основан на демонстрационных файлах:

```text
examples/ru/simple-landing/project-questionnaire.example.md
examples/ru/simple-landing/legal-documents-matrix.example.md
```

Все данные ниже демонстрационные. Перед публикацией документов и изменениями интерфейса нужно провести юридическую и техническую проверку.

---

## 1. Что внедряем

| Элемент | Статус | Комментарий |
|---|---|---|
| Публичные страницы документов | нужно | Документы должны открываться без авторизации |
| Ссылки на документы в футере | нужно | Минимум privacy, personal_data_consent, cookie_policy |
| Согласие под формой заявки | нужно | Чекбокс или понятный текст согласия |
| Отдельное рекламное согласие | уточнить | Нужно, если есть рассылка или маркетинговые сообщения |
| Cookie-уведомление или баннер | уточнить | Зависит от cookies, аналитики, пикселей и юрисдикции |
| Описание передачи заявок в CRM | уточнить | Нужно, если заявки уходят в CRM или внешнюю систему |
| Описание email/SMS/мессенджер-уведомлений | уточнить | Нужно, если после заявки отправляются сообщения |
| Хранение факта согласия | желательно | Дата, текст согласия, версия документа, IP, user-agent |
| Страница «Правовые документы» | желательно | Удобная точка входа для всех документов |

---

## 2. Публичные URL документов

Рекомендуемые URL для простого лендинга:

| Документ | URL | Нужен |
|---|---|---|
| Политика конфиденциальности | `/legal/privacy` | да |
| Согласие на обработку персональных данных | `/legal/personal-data-consent` | да |
| Политика cookies | `/legal/cookie-policy` | если есть cookies/аналитика/пиксели |
| Согласие на рекламные сообщения | `/legal/marketing-consent` | если есть рекламная рассылка |
| Правила поддержки | `/legal/support-rules` | если заявлены сроки/условия поддержки |
| Правила использования бота | `/legal/bot-terms` | если есть полноценный Telegram/MAX-бот |
| Публичная оферта | `/legal/offer` | если появится онлайн-оплата |
| Правила оплаты и возврата | `/legal/payment-and-refund` | если появится онлайн-оплата |
| Правила подписки | `/legal/subscription-rules` | если появится подписка/trial |

Если сайт статический, документы можно разместить как HTML-файлы:

```text
/public/legal/privacy.html
/public/legal/personal-data-consent.html
/public/legal/cookie-policy.html
```

## 3. Футер сайта

### Что добавить

| Ссылка | Нужна | Комментарий |
|---|---|---|
| Политика конфиденциальности | да | При любой форме заявки или аналитике |
| Согласие на обработку персональных данных | да | При сборе телефона/email/имени |
| Политика cookies | если используются cookies | Аналитика, пиксели, localStorage |
| Согласие на рекламные сообщения | если есть рассылка | Можно добавить рядом с формой подписки |
| Контакты | да | Email, телефон, мессенджер или форма |
| Правовые документы | желательно | Общая страница со всеми ссылками |

### Что проверить

| Проверка | Статус | Комментарий |
|---|---|---|
| Ссылки видны на десктопе | [ ] | Не спрятаны в мелком тексте |
| Ссылки видны на мобильной версии | [ ] | Проверить адаптивность |
| Ссылки открываются без авторизации | [ ] | Для лендинга это обязательно |
| Ссылки не ведут на 404 | [ ] | Проверить вручную |
| Документы содержат дату редакции и версию | [ ] | Для прозрачности и фиксации согласий |
| В документах нет плейсхолдеров | [ ] | `[УТОЧНИТЬ]`, `[ИНН]`, `[EMAIL]` |

---

## 4. Форма заявки

### Что проверить

| Проверка | Статус | Комментарий |
|---|---|---|
| В форме указаны только необходимые поля | [ ] | Не собирать лишние данные |
| Обязательные поля отмечены явно | [ ] | Например: имя, телефон/email |
| Под формой есть ссылка на Политику конфиденциальности | [ ] | До отправки формы |
| Под формой есть согласие на обработку персональных данных | [ ] | Чекбокс или явный текст |
| Если используется чекбокс, он не отмечен заранее | [ ] | Пользователь должен сам выразить согласие |
| Без обязательного согласия форма не отправляется | [ ] | Проверить frontend и backend |
| Ошибка при отсутствии согласия понятна | [ ] | Например: «Необходимо дать согласие» |
| Заявка не отправляет лишние данные в аналитику | [ ] | Не передавать телефон/email в события без необходимости |
| Заявка сохраняет источник/UTM безопасно | [ ] | Без персональных данных в URL |
| После отправки пользователь видит понятное сообщение | [ ] | Что заявка принята и как с ним свяжутся |

### Рекомендуемый вариант с чекбоксом

```text
[ ] Я даю согласие на обработку персональных данных для обработки моей заявки и обратной связи
```

Текст должен содержать ссылки на:

```text
Политику конфиденциальности
Согласие на обработку персональных данных
```

### Упрощенный вариант без чекбокса

```text
Нажимая «Отправить», я даю согласие на обработку персональных данных для обработки моего обращения и соглашаюсь с Политикой конфиденциальности.
```

Для более надежного сценария лучше использовать отдельный чекбокс.

## 5. Форма обратной связи

Если форма обратной связи отличается от формы заявки, ее нужно проверить отдельно.

| Проверка | Статус | Комментарий |
|---|---|---|
| Указана цель обработки данных | [ ] | Например: ответ на обращение |
| Есть ссылка на Политику конфиденциальности | [ ] | До отправки формы |
| Есть согласие на обработку персональных данных | [ ] | Чекбокс или текст |
| Форма не подписывает пользователя на рекламу автоматически | [ ] | Нужен отдельный чекбокс |
| Форма не собирает лишние данные | [ ] | Минимизация данных |
| Есть защита от спама | [ ] | Капча, rate limit или другой механизм |
| Капча описана в privacy/cookie_policy, если используется | [ ] | Капча может передавать технические данные |

## 6. Форма подписки на новости

Если на лендинге есть подписка на новости, полезные материалы или рекламные предложения.

| Проверка | Статус | Комментарий |
|---|---|---|
| Есть отдельный документ marketing_consent | [ ] | Если сообщения рекламные/маркетинговые |
| Чекбокс рекламного согласия не отмечен заранее | [ ] | Пользователь должен согласиться сам |
| Подписка на рекламу не обязательна для заявки | [ ] | Отдельная цель обработки |
| Есть ссылка на Политику конфиденциальности | [ ] | До отправки формы |
| Есть способ отписки | [ ] | Ссылка в письме, команда, обращение |
| Отзыв согласия фиксируется | [ ] | Если есть журнал согласий |
| Email-провайдер или сервис рассылок указан в privacy | [ ] | Как третье лицо или обработчик |

### Рекомендуемый текст

```text
[ ] Я согласен получать информационные и рекламные сообщения
```

### Подтверждение после подписки

```text
Спасибо! Вы подписались на новости. Отписаться можно по ссылке в письме или через обращение в поддержку.
```

## 7. Cookie-баннер и аналитика

### Что проверить

| Проверка | Статус | Комментарий |
|---|---|---|
| Составлен список cookies и счетчиков | [ ] | Технические, аналитические, рекламные |
| Подготовлена Политика cookies | [ ] | Если cookies/аналитика используются |
| В privacy описаны технические данные | [ ] | IP, user-agent, cookies, UTM |
| Указаны системы аналитики | [ ] | Яндекс.Метрика, Google Analytics, Top.Mail.Ru |
| Указаны рекламные пиксели | [ ] | Если есть ретаргетинг/реклама |
| Есть cookie-уведомление или баннер | [ ] | Если выбран такой сценарий |
| Баннер содержит ссылку на cookie policy | [ ] | Пользователь должен открыть документ |
| Необязательные cookies не ставятся до согласия | [ ] | Если проект выбрал строгую модель согласия |
| События аналитики не содержат телефон/email | [ ] | Важная проверка безопасности |
| UTM-метки не содержат персональные данные | [ ] | Не передавать email/телефон в UTM |

### Простой текст cookie-уведомления

```text
Мы используем cookies и похожие технологии для работы сайта, аналитики и улучшения сервиса. Подробнее — в Политике cookies.
```

### Текст для сценария с выбором

```text
Мы используем необходимые cookies для работы сайта, а также аналитические и рекламные cookies с вашего согласия. Вы можете принять все cookies, отклонить необязательные или настроить выбор.
```

## 8. CRM и передача заявок

Если заявки с лендинга уходят в CRM, email, таблицу, Telegram/MAX-бота или иную систему.

| Проверка | Статус | Комментарий |
|---|---|---|
| Определено, куда отправляются заявки | [ ] | CRM, email, таблица, бот, webhook |
| Передача заявок описана в privacy | [ ] | Третьи лица / обработчики |
| CRM-провайдер указан в документах | [ ] | Если используется внешняя CRM |
| Email-провайдер указан в документах | [ ] | Если заявки уходят по email |
| В логах не сохраняется лишняя персональная информация | [ ] | Не дублировать данные без необходимости |
| В webhook нет секретов в URL | [ ] | Не передавать токены в query-параметрах |
| Ошибки CRM/webhook не показываются пользователю с техническими деталями | [ ] | Без stack trace и токенов |
| Есть резервный сценарий при ошибке отправки заявки | [ ] | Лог, повтор, email, уведомление администратору |

## 9. Кнопка перехода в мессенджер

Если на лендинге есть кнопка «Написать в Telegram/MAX/WhatsApp».

| Проверка | Статус | Комментарий |
|---|---|---|
| Пользователь понимает, что переходит во внешний сервис | [ ] | Особенно если открывается мессенджер |
| В privacy описано, что обращение через мессенджер обрабатывается отдельно | [ ] | Если владелец получает сообщения |
| Если это просто ссылка на чат — bot_terms может быть не нужен | [ ] | Достаточно privacy |
| Если это полноценный бот — нужен bot_terms | [ ] | Команды, уведомления, регистрация, данные мессенджера |
| Если бот собирает данные — нужно согласие в боте | [ ] | Не только на лендинге |
| Если через бот идет реклама — нужен marketing_consent | [ ] | Отдельное согласие |

## 10. Онлайн-чат и капча

Если на лендинге используются сторонние виджеты.

| Виджет | Что проверить |
|---|---|
| Онлайн-чат | Указать провайдера в privacy/cookie_policy, описать сообщения и технические данные |
| Капча | Указать передачу IP, user-agent и технических данных, если используется сторонний сервис |
| Встроенное видео | Указать сторонний сервис и cookies, если применимо |
| Карта | Указать сторонний сервис и cookies, если применимо |
| Виджет календаря | Указать передачу данных при записи на встречу |

## 11. Хранение согласий

Для простого лендинга можно хранить минимальный набор данных.

| Сценарий | Что хранить | Источник |
|---|---|---|
| Отправка формы заявки | checkbox_text, document_version, accepted_at, ip_address, user_agent, source_url | landing_form |
| Обратная связь | checkbox_text, document_version, accepted_at, ip_address, user_agent, source_url | feedback_form |
| Подписка на новости | marketing_consent, accepted_at, source_url, email_hash | subscription_form |
| Отписка от рассылки | event_type=revoked, revoked_at, source | unsubscribe_link |
| Принятие cookies | categories, policy_version, accepted_at/rejected_at | cookie_banner |

## 14. Проверка безопасности

| Проверка | Статус | Комментарий |
|---|---|---|
| Форма защищена от спама | [ ] | Капча, rate limit, honeypot |
| Валидация есть на frontend и backend | [ ] | Нельзя полагаться только на frontend |
| Ошибки формы не раскрывают внутренние данные | [ ] | Без stack trace |
| В URL не передаются персональные данные | [ ] | Телефон/email нельзя в query |
| В события аналитики не передаются персональные данные | [ ] | Проверить goals/events |
| API-ключи CRM/email не попали в frontend | [ ] | Только backend/env |
| Логи не содержат секреты | [ ] | Токены, ключи, пароли |
| Доступ к заявкам ограничен | [ ] | Админка, email, CRM |

## 16. Итоговый вывод агента

Для Demo Landing нужно внедрить правовые документы в три основные точки:

1. Футер сайта и публичные страницы документов.
2. Формы заявки/обратной связи с согласием на обработку персональных данных.
3. Cookie-уведомление и описание аналитики, если используются cookies, счетчики или рекламные пиксели.

Минимально нужно добавить:

- `/legal/privacy`;
- `/legal/personal-data-consent`;
- `/legal/cookie-policy`, если есть cookies/аналитика;
- ссылки в футере;
- согласие под формой заявки;
- безопасную фиксацию факта согласия;
- описание CRM, email-провайдера, аналитики и рекламных систем в privacy/cookie_policy.

Если есть рекламная рассылка, полноценный бот, поддержка с условиями или онлайн-оплата, нужно добавить соответствующие документы и интерфейсные подтверждения.

Документы, интерфейс и хранение согласий требуют проверки владельцем проекта, техническим специалистом и юристом перед публикацией.
