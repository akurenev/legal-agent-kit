

# Contributing to Legal Agent Kit

Спасибо за интерес к **Legal Agent Kit**.

Этот репозиторий создан как открытый набор инструкций, шаблонов, чек-листов, примеров и промптов для AI-агентов, которые помогают готовить черновики правовых документов для цифровых проектов.

Мы приветствуем вклад, который помогает сделать kit понятнее, безопаснее, точнее и полезнее для разработчиков, предпринимателей, продуктовых команд и AI-агентов.

---

## Важный дисклеймер

Legal Agent Kit не является юридической консультацией и не заменяет работу юриста.

Любые шаблоны, чек-листы и промпты в этом репозитории предназначены для:

- структурирования информации о проекте;
- подготовки черновиков документов;
- выявления вопросов и рисков;
- помощи владельцу проекта перед юридической проверкой.

Не добавляйте материалы, которые создают впечатление, что документы можно публиковать без проверки юристом.

---

## Какой вклад полезен

Вы можете предложить:

- улучшения `README.md`;
- улучшения `AGENTS.md`;
- дополнения к `LEGAL_AGENT_GUIDE.md`;
- новые вопросы для `LEGAL_PROJECT_QUESTIONNAIRE.md`;
- новые шаблоны документов;
- улучшения существующих шаблонов;
- новые чек-листы;
- новые примеры для типовых проектов;
- новые промпты для Codex / ChatGPT / других AI-агентов;
- исправления опечаток;
- улучшения структуры и навигации;
- уточнения по интерфейсному внедрению документов;
- рекомендации по хранению согласий;
- безопасные формулировки дисклеймеров;
- переводы на другие языки, если они будут поддерживаться.

---

## Что не стоит добавлять

Не добавляйте в репозиторий:

- чужие документы, скопированные с сайтов без разрешения;
- персональные данные реальных пользователей;
- реальные API-токены, ключи, пароли, секреты, webhook secret;
- реальные банковские данные;
- реальные договоры клиентов без явного разрешения и обезличивания;
- утверждения о гарантированном соответствии закону;
- формулировки «документ готов к публикации» без юридической проверки;
- агрессивные или вводящие в заблуждение юридические обещания;
- шаблоны, завязанные на одну компанию без возможности адаптации;
- код, который автоматически публикует документы без проверки владельцем проекта.

## Общие принципы

При подготовке изменений соблюдайте следующие принципы:

1. **Не выдумывать данные.** Если данные неизвестны, используйте плейсхолдер `[УТОЧНИТЬ]`.
2. **Не обещать юридическую точность.** Любой результат должен требовать проверки владельцем проекта и юристом.
3. **Описывать фактическую работу проекта.** Документы должны соответствовать реальной функциональности, а не маркетинговому описанию.
4. **Разделять согласия.** Согласие на обработку персональных данных, рекламные сообщения, оферта и пользовательское соглашение не должны смешиваться без необходимости.
5. **Учитывать интерфейс.** Документы должны быть связаны с реальными точками: регистрация, заявка, оплата, бот, API, интеграции.
6. **Сохранять понятный язык.** Тексты должны быть структурированными и читаемыми для обычного пользователя.
7. **Думать о безопасности.** Не допускайте логирования токенов, паролей, банковских секретов и лишних персональных данных.
8. **Оставлять вопросы.** Если условие зависит от бизнес-модели, юрисдикции или фактической реализации, добавляйте вопрос к владельцу проекта или юристу.

## Стиль Markdown

Используйте простой Markdown:

- заголовки через `#`, `##`, `###`;
- таблицы для матриц и чек-листов;
- списки для условий и требований;
- кодовые блоки для примеров команд, URL, checkbox-текстов и структур данных;
- плейсхолдеры в квадратных скобках: `[УТОЧНИТЬ]`, `[EMAIL]`, `[ДОМЕН]`.

Не используйте сложное оформление, HTML-разметку и нестандартные расширения Markdown без необходимости.

## Именование файлов

Сохраняйте текущий стиль именования.

### Шаблоны документов

Шаблоны на русском языке находятся в:

```text
templates/ru/
```

Используйте понятные имена:

```text
terms.template.md
privacy.template.md
personal-data-consent.template.md
offer.template.md
payment-and-refund.template.md
cookie-policy.template.md
support-rules.template.md
data-processing.template.md
subscription-rules.template.md
marketing-consent.template.md
bank_data_processing.md
api_terms.md
bot_terms.md
```

Если добавляете новый шаблон, укажите его в:

- `README.md`;
- `AGENTS.md`;
- `LEGAL_AGENT_GUIDE.md`, если он влияет на методику;
- `docs/document-matrix.md`, если меняется логика выбора документов.

### Чек-листы

Чек-листы находятся в:

```text
checklists/ru/
```

Имена файлов должны быть понятными:

```text
launch-checklist.md
registration-form-checklist.md
payment-page-checklist.md
subscription-checklist.md
personal-data-checklist.md
cookies-and-analytics-checklist.md
api-tokens-checklist.md
```

### Примеры

Примеры находятся в:

```text
examples/ru/
```

Для каждого примера желательно иметь:

```text
project-questionnaire.example.md
legal-documents-matrix.example.md
legal-integration-checklist.example.md
```

Если пример не требует всех трех файлов, объясните это в PR.

## Требования к шаблонам документов

Каждый шаблон документа должен содержать:

- название документа;
- дату редакции;
- версию;
- название сервиса;
- домен;
- дисклеймер о необходимости адаптации и юридической проверки;
- раздел с общими положениями;
- понятные определения, если они нужны;
- разделы, зависящие от фактической функциональности;
- контакты владельца сервиса;
- связанные документы;
- плейсхолдеры для неизвестных данных.

Рекомендуемый верхний блок:

```text
# [НАЗВАНИЕ ДОКУМЕНТА]

Редакция от: [ДАТА]
Версия: [ВЕРСИЯ]
Сервис: [НАЗВАНИЕ СЕРВИСА]
Домен: [ДОМЕН]

> Этот документ является шаблоном и должен быть адаптирован под конкретный проект. Перед публикацией документ необходимо проверить с юристом.
```

---

## Требования к чек-листам

Чек-лист должен помогать агенту или владельцу проекта проверить конкретный сценарий.

Хороший чек-лист содержит:

- назначение;
- когда использовать;
- таблицу проверок;
- типовые ошибки;
- финальную проверку;
- итоговый вывод агента.

Для чекбоксов используйте формат:

```md
| Проверка | Статус | Комментарий |
|---|---|---|
| Документы доступны без авторизации | [ ] | Проверить вручную |
```

---

## Требования к примерам

Примеры должны быть демонстрационными, без реальных персональных данных и без реальных реквизитов.

Используйте безопасные значения:

```text
Demo SaaS
Demo Bot Service
Demo Landing
https://example.com
support@example.com
[УТОЧНИТЬ]
```

Пример должен показывать ход мысли агента:

1. Что это за проект.
2. Какие функции есть.
3. Какие данные собираются.
4. Какие документы нужны.
5. Какие вопросы нужно уточнить.
6. Какие риски есть.
7. Что нужно внедрить в интерфейс.

---

## Требования к промптам

Промпты в `prompts/` должны быть пригодны для прямого использования в Codex или другом AI-агенте.

Промпт должен содержать:

- назначение;
- основной промпт;
- ограничения;
- порядок работы;
- ожидаемые файлы результата;
- формат финального ответа;
- короткую версию промпта, если файл большой.

Промпт не должен требовать от агента сразу менять рабочий код без анализа и подтверждения пользователя.

## Как предлагать изменения

Рекомендуемый процесс:

1. Сделайте fork репозитория.
2. Создайте отдельную ветку:

```bash
git checkout -b feature/your-change-name
```

3. Внесите изменения.
4. Проверьте структуру Markdown.
5. Убедитесь, что не добавили реальные персональные данные или секреты.
6. Сделайте commit:

```bash
git add .
git commit -m "Describe your change"
```

7. Отправьте ветку:

```bash
git push origin feature/your-change-name
```

8. Создайте Pull Request.

## Формат Pull Request

В Pull Request желательно указать:

```md
## Что изменено

- [краткое описание]

## Тип изменения

- [ ] Документация
- [ ] Новый шаблон
- [ ] Новый чек-лист
- [ ] Новый пример
- [ ] Новый промпт
- [ ] Исправление ошибки/опечатки
- [ ] Другое

## Почему это полезно

[описание]

## Проверка

- [ ] Нет реальных персональных данных
- [ ] Нет токенов, паролей и секретов
- [ ] Есть дисклеймер о юридической проверке
- [ ] Плейсхолдеры используются вместо неизвестных данных
- [ ] README / AGENTS / guide обновлены, если структура изменилась
```

## Commit messages

Пишите короткие и понятные сообщения коммитов.

Примеры:

```text
Add bot terms template
Add simple landing legal examples
Update README structure
Fix typo in consent checklist
Improve API tokens checklist
```

---

## Проверка перед отправкой PR

Перед отправкой Pull Request проверьте:

- [ ] Markdown открывается и читается нормально;
- [ ] нет реальных персональных данных;
- [ ] нет API-токенов, паролей, ключей и секретов;
- [ ] новые файлы указаны в README, если это нужно;
- [ ] новые файлы указаны в AGENTS.md, если это нужно;
- [ ] в шаблонах есть дата редакции и версия;
- [ ] в шаблонах есть дисклеймер;
- [ ] неизвестные данные оставлены как `[УТОЧНИТЬ]`;
- [ ] рекламное согласие не сделано обязательным без необходимости;
- [ ] документы не обещают юридическую финальность;
- [ ] изменения не противоречат существующим шаблонам и чек-листам.

---

## Работа с юридическими формулировками

Если вы предлагаете юридически чувствительную формулировку, добавьте комментарий в Pull Request:

- для какого типа проекта она нужна;
- какие риски закрывает;
- где ее лучше использовать;
- требует ли проверки юристом;
- какие данные нужно уточнить у владельца проекта.

Не добавляйте категоричные утверждения о соответствии конкретному закону без возможности проверки.

---

## Работа с персональными данными и секретами

Никогда не добавляйте в репозиторий:

- реальные ФИО пользователей;
- реальные телефоны;
- реальные email клиентов;
- реальные адреса;
- реальные банковские реквизиты клиентов;
- реальные API-токены;
- реальные OAuth-токены;
- реальные webhook secret;
- реальные платежные secret key;
- дампы баз данных;
- логи с персональными данными;
- скриншоты личных кабинетов с данными пользователей.

Если пример требует данных, используйте:

```text
[УТОЧНИТЬ]
example.com
support@example.com
Demo User
Demo Company
```

---

## Лицензия вклада

Отправляя Pull Request, вы подтверждаете, что ваш вклад может быть распространен на условиях лицензии репозитория.

Если вы добавляете материалы, созданные не вами, укажите источник и убедитесь, что их можно использовать в этом репозитории.

---

## Вопросы и обсуждения

Если вы не уверены, как лучше оформить изменение:

- создайте Issue;
- опишите сценарий проекта;
- укажите, какой документ или чек-лист хотите улучшить;
- предложите вариант формулировки;
- отметьте спорные места.

Лучше сначала обсудить спорную юридическую или продуктовую логику, чем добавлять универсальную формулировку, которая может не подходить большинству проектов.
