

# LEGAL_AGENT_GUIDE.md

## Назначение

`LEGAL_AGENT_GUIDE.md` — главное методическое руководство для AI-агента, который готовит черновики правовых документов для цифрового проекта.

Документ описывает:

- какие проекты поддерживаются;
- какие данные нужно собрать;
- как выбрать комплект документов;
- как работать с шаблонами;
- как формировать плейсхолдеры и вопросы;
- как подготовить рекомендации по внедрению документов в интерфейс;
- как выполнить финальную проверку результата.

Материалы, подготовленные по этому руководству, являются черновиками и требуют юридической проверки перед публикацией.

---

## 1. Для каких проектов использовать

Руководство подходит для подготовки документов для следующих типов проектов:

- лендинг;
- корпоративный сайт;
- интернет-магазин;
- SaaS-сервис;
- PWA-приложение;
- Telegram-бот;
- MAX-бот;
- мобильное приложение;
- API-сервис;
- сервис подписки;
- сервис уведомлений;
- сервис обработки данных;
- маркетплейс-интеграция;
- сервис с банковскими или платежными интеграциями;
- внутренний B2B-сервис с доступом клиентов;
- облачный сервис с несколькими клиентами и пользователями.

---

## 2. Главный принцип работы агента

Агент не должен создавать универсальный набор документов «на всякий случай».

Сначала нужно понять фактическую работу проекта, а затем подобрать документы под реальные функции сервиса.

Агент должен учитывать:

- что делает сервис;
- кто им пользуется;
- есть ли регистрация;
- есть ли личный кабинет;
- есть ли оплата;
- есть ли подписка;
- есть ли пробный период;
- есть ли автопродление;
- какие данные собираются;
- используются ли cookies и аналитика;
- есть ли API, webhook и интеграции;
- передает ли пользователь токены или ключи доступа;
- получает ли сервис банковские или финансовые данные;
- есть ли боты и уведомления;
- есть ли пользовательский контент;
- от каких внешних сервисов зависит работа проекта.

---

## 3. Обязательный порядок работы

Агент должен выполнять работу в такой последовательности:

1. Изучить описание проекта.
2. Изучить доступные файлы проекта:
   - `README.md`;
   - `AGENTS.md`;
   - `.env.example`;
   - маршруты API;
   - страницы регистрации;
   - страницы оплаты;
   - личный кабинет;
   - футер сайта;
   - тексты тарифов;
   - существующие правовые документы.
3. Заполнить `LEGAL_PROJECT_QUESTIONNAIRE.md`.
4. Определить тип проекта и модель работы: B2B, B2C или смешанная.
5. Выбрать комплект документов по матрице.
6. Подготовить документы на основе шаблонов из `templates/ru/`.
7. Заменить известные плейсхолдеры фактическими данными проекта.
8. Неизвестные данные оставить как плейсхолдеры.
9. Сформировать список плейсхолдеров.
10. Сформировать список вопросов для владельца проекта или юриста.
11. Сформировать список юридических рисков.
12. Сформировать рекомендации по внедрению документов в интерфейс.
13. Сформировать рекомендации по хранению согласий пользователей.
14. Выполнить финальную проверку документов.

## 4. Какие данные нужно собрать

Перед подготовкой документов агент должен собрать данные по следующим блокам.

### 4.1. Владелец проекта

- полное наименование владельца;
- организационно-правовая форма;
- ИНН;
- ОГРН / ОГРНИП;
- юридический адрес;
- почтовый адрес;
- email для обращений;
- телефон;
- сайт;
- контактное лицо.

Если эти данные неизвестны, агент должен оставить плейсхолдеры и добавить вопросы в список уточнений.

### 4.2. Описание проекта

- название сервиса;
- домен;
- краткое описание;
- тип проекта;
- основная функция;
- целевая аудитория;
- география работы;
- юрисдикция;
- язык документов.

### 4.3. Пользователи

- физические лица;
- ИП;
- юридические лица;
- сотрудники клиентов;
- администраторы клиентов;
- внутренние сотрудники сервиса;
- роли и уровни доступа;
- возможность удаления аккаунта;
- возможность блокировки пользователя администрацией.

### 4.4. Регистрация и авторизация

- способ регистрации;
- способ авторизации;
- подтверждение email;
- подтверждение телефона;
- восстановление пароля;
- авторизация через внешние сервисы;
- фиксация согласий при регистрации.

### 4.5. Персональные данные

- ФИО;
- email;
- телефон;
- адрес;
- IP-адрес;
- user-agent;
- cookies;
- Telegram ID;
- MAX ID;
- данные организации;
- платежные данные;
- банковские данные;
- API-токены;
- данные из внешних сервисов;
- иные данные.

Для каждого вида данных нужно определить:

- собираются ли они;
- цель обработки;
- являются ли обязательными;
- срок хранения;
- кому передаются;
- можно ли их удалить.

### 4.6. Платежи и подписки

- есть ли платные услуги;
- есть ли публичные тарифы;
- есть ли индивидуальные тарифы;
- есть ли подписка;
- период подписки;
- пробный период;
- автопродление;
- льготный период;
- платежный провайдер;
- фискализация;
- выдача чека;
- момент оказания услуги;
- условия возврата;
- порядок отключения подписки.

### 4.7. Интеграции

- банки;
- платежные сервисы;
- маркетплейсы;
- мессенджеры;
- CRM;
- сервисы аналитики;
- email/SMS/push-провайдеры;
- внешние API;
- webhook;
- поставщики данных.

Для каждой интеграции нужно определить:

- какие данные передаются;
- какие данные получаются;
- кто является стороной интеграции;
- что происходит при сбое внешнего сервиса;
- кто отвечает за корректность переданных пользователем токенов и ключей.

## 5. Матрица выбора документов

Агент должен выбирать документы по фактическим функциям проекта.

| Условие | Документы |
|---|---|
| Есть сайт без регистрации и оплаты | Политика конфиденциальности, Политика cookies при наличии cookies/аналитики |
| Есть форма обратной связи | Политика конфиденциальности, Согласие на обработку персональных данных |
| Есть регистрация | Пользовательское соглашение, Политика конфиденциальности, Согласие на обработку персональных данных |
| Есть личный кабинет | Пользовательское соглашение, Политика конфиденциальности, Согласие на обработку персональных данных |
| Есть онлайн-оплата | Публичная оферта, Правила оплаты и возврата |
| Есть подписка | Публичная оферта, Правила подписки, Правила оплаты и возврата |
| Есть пробный период | Публичная оферта, Правила подписки |
| Есть автопродление | Публичная оферта, Правила подписки, отдельное описание условий списаний |
| Есть B2C-пользователи | Условия для потребителей, правила возврата, информация об исполнителе |
| Есть B2B-клиенты | Условия для юридических лиц и ИП, порядок доступа сотрудников клиента |
| Есть cookies или аналитика | Политика cookies |
| Есть рекламные рассылки | Согласие на информационные и рекламные сообщения |
| Есть Telegram/MAX-бот | Раздел о работе через мессенджер, уведомлениях и идентификаторах пользователя |
| Есть API | Раздел об API, токенах, ограничениях и ответственности |
| Есть webhook | Раздел о доставке событий, сбоях и ответственности за принимающую сторону |
| Пользователь передает API-токены | Документ или раздел об обработке токенов и ключей доступа |
| Сервис получает банковские данные | Документ или раздел об обработке банковских/финансовых данных |
| Есть платежный провайдер | Раздел о передаче данных платежному провайдеру |
| Есть внешние поставщики | Раздел об ограничениях ответственности за внешние сервисы |
| Есть пользовательский контент | Правила размещения, удаления и модерации контента |
| Есть поддержка | Правила поддержки, SLA при необходимости |

## 6. Рекомендуемый комплект документов

### 6.1. Минимальный сайт или лендинг

Если проект — простой сайт без регистрации и оплаты, обычно нужны:

- Политика конфиденциальности;
- Согласие на обработку персональных данных, если есть формы;
- Политика cookies, если используются cookies, аналитика или рекламные пиксели.

### 6.2. SaaS с регистрацией

Если есть регистрация и личный кабинет:

- Пользовательское соглашение;
- Политика конфиденциальности;
- Согласие на обработку персональных данных;
- Политика cookies;
- Правила поддержки, если есть поддержка пользователей.

### 6.3. SaaS с оплатой и подпиской

Если есть платные тарифы, подписка или пробный период:

- Пользовательское соглашение;
- Политика конфиденциальности;
- Согласие на обработку персональных данных;
- Публичная оферта;
- Правила оплаты и возврата;
- Правила подписки;
- Политика cookies;
- Правила поддержки.

### 6.4. Бот в Telegram или MAX

Если проект работает через мессенджер:

- Пользовательское соглашение;
- Политика конфиденциальности;
- Согласие на обработку персональных данных;
- Правила использования бота;
- Правила уведомлений;
- Оферта, если есть платные функции.

### 6.5. API-сервис

Если сервис предоставляет API:

- Пользовательское соглашение;
- Условия использования API;
- Политика конфиденциальности;
- Правила обработки токенов;
- Правила поддержки;
- Оферта, если API платный.

### 6.6. Маркетплейс-интеграция

Если сервис подключается к Ozon, Яндекс Маркет, Wildberries или другим платформам:

- Пользовательское соглашение;
- Политика конфиденциальности;
- Согласие на обработку персональных данных;
- Документ или раздел об обработке API-токенов;
- Раздел об ограничениях ответственности за работу маркетплейса;
- Оферта и правила оплаты, если сервис платный.

### 6.7. Сервис с банковскими данными

Если сервис получает выписки, операции, остатки по счетам или иные финансовые данные:

- Пользовательское соглашение;
- Политика конфиденциальности;
- Согласие на обработку персональных данных;
- Документ или раздел об обработке банковских данных;
- Раздел о том, что сервис не является финансовым, налоговым или юридическим консультантом, если применимо;
- Оферта и правила оплаты, если сервис платный.

## 7. Требования к документам

### 7.1. Общие требования

Каждый документ должен содержать:

- название документа;
- дату редакции;
- номер версии;
- название сервиса;
- сведения о владельце сервиса или плейсхолдеры;
- область применения;
- понятную структуру;
- контакты для обращений;
- порядок изменения документа.

Рекомендуемый блок в начале документа:

```text
Редакция от: [ДАТА]
Версия: [ВЕРСИЯ]
Сервис: [НАЗВАНИЕ СЕРВИСА]
```

### 7.2. Пользовательское соглашение

Документ должен описывать:

- термины;
- предмет соглашения;
- порядок регистрации;
- порядок использования сервиса;
- права и обязанности пользователя;
- права и обязанности администрации;
- ограничения использования;
- блокировку и удаление аккаунта;
- интеллектуальные права;
- уведомления;
- ответственность сторон;
- внешние сервисы;
- изменение условий;
- контакты.

### 7.3. Политика конфиденциальности

Документ должен описывать:

- оператора данных;
- какие данные собираются;
- цели обработки;
- правовые основания обработки;
- действия с данными;
- сроки хранения;
- передачу третьим лицам;
- используемые сервисы;
- защиту данных;
- права пользователя;
- порядок отзыва согласия;
- контакты.

### 7.4. Согласие на обработку персональных данных

Документ должен быть отдельным от пользовательского соглашения.

Он должен содержать:

- кто дает согласие;
- кому дается согласие;
- перечень данных;
- цели обработки;
- действия с данными;
- срок действия согласия;
- порядок отзыва;
- ссылку на политику конфиденциальности.

### 7.5. Публичная оферта

Документ должен описывать:

- исполнителя;
- пользователя или заказчика;
- предмет договора;
- порядок акцепта;
- состав услуги;
- тарифы;
- порядок оплаты;
- порядок оказания услуги;
- подписку и сроки доступа;
- пробный период;
- автопродление, если есть;
- возвраты;
- ограничения;
- ответственность сторон;
- изменение тарифов;
- расторжение договора;
- контакты.

### 7.6. Правила оплаты и возврата

Документ должен описывать:

- способы оплаты;
- платежного провайдера;
- момент оплаты;
- выдачу чека;
- начало оказания услуги;
- условия возврата;
- случаи, когда возврат невозможен или ограничен;
- порядок обращения за возвратом;
- срок рассмотрения обращения;
- контакты.

### 7.7. Политика cookies

Документ должен описывать:

- что такое cookies;
- какие cookies используются;
- цели использования;
- технические cookies;
- аналитические cookies;
- рекламные cookies;
- сторонние сервисы аналитики;
- как пользователь может отключить cookies.

### 7.8. Правила подписки

Документ должен описывать:

- доступные тарифы;
- период подписки;
- пробный период;
- порядок подключения;
- порядок продления;
- автопродление, если есть;
- льготный период, если есть;
- ограничения тарифа;
- порядок отключения;
- последствия окончания подписки;
- правила возврата.

### 7.9. Правила поддержки

Документ должен описывать:

- каналы поддержки;
- время работы поддержки;
- категории обращений;
- сроки реакции, если они заявлены;
- что входит в поддержку;
- что не входит в поддержку;
- платную поддержку, если есть;
- ограничения ответственности.

### 7.10. Документ об обработке API-токенов и данных интеграций

Документ или раздел должен описывать:

- какие токены пользователь передает сервису;
- для чего они используются;
- кто отвечает за корректность токенов;
- как токены хранятся;
- можно ли удалить токены;
- какие внешние сервисы вызываются;
- какие данные передаются во внешние сервисы;
- что происходит при сбоях внешних сервисов;
- ограничения ответственности сервиса.

### 7.11. Документ об обработке банковских данных

Документ или раздел должен описывать:

- через какой механизм подключается банк;
- какие банковские данные получает сервис;
- получает ли сервис выписки, операции, остатки, данные контрагентов;
- может ли сервис инициировать платежи;
- работает ли сервис только на чтение;
- где и сколько хранятся банковские данные;
- можно ли отключить банк и удалить данные;
- кому могут передаваться данные;
- является ли сервис финансовым, налоговым или юридическим консультантом;
- ограничения ответственности за решения пользователя на основе данных сервиса.

## 9. Рекомендации по внедрению в интерфейс

Агент должен подготовить рекомендации для разработчика.

### 9.1. Футер сайта

В футере сайта рекомендуется разместить ссылки:

- Пользовательское соглашение;
- Политика конфиденциальности;
- Согласие на обработку персональных данных;
- Публичная оферта;
- Правила оплаты и возврата;
- Политика cookies;
- Контакты.

Набор ссылок зависит от проекта.

### 9.2. Форма регистрации

Для проекта с регистрацией рекомендуется использовать отдельные чекбоксы:

```text
[ ] Я принимаю Пользовательское соглашение
[ ] Я ознакомлен с Политикой конфиденциальности
[ ] Я даю согласие на обработку персональных данных
```

Если есть рекламные рассылки:

```text
[ ] Я согласен получать информационные и рекламные сообщения
```

Согласие на рекламные сообщения не должно быть обязательным, если оно не является необходимым для оказания услуги.

### 9.3. Страница оплаты

Перед оплатой рекомендуется разместить текст:

```text
Нажимая «Оплатить», я принимаю условия Публичной оферты и Правила оплаты и возврата.
```

Если есть подписка:

```text
Нажимая «Оплатить», я принимаю условия Публичной оферты, Правила подписки и Правила оплаты и возврата.
```

### 9.4. Личный кабинет

В личном кабинете рекомендуется добавить раздел:

```text
Правовые документы
```

В нем можно разместить:

- актуальные редакции документов;
- дату редакции;
- историю согласий пользователя;
- возможность скачать документы.

## 10. Хранение согласий

Если проект содержит регистрацию, личный кабинет, оплату, подписку или обработку персональных данных, агент должен предложить хранить факты согласия пользователей.

Рекомендуемые поля:

| Поле | Описание |
|---|---|
| `id` | ID записи |
| `user_id` | ID пользователя |
| `document_type` | Тип документа |
| `document_version` | Версия документа |
| `document_date` | Дата редакции документа |
| `document_url` | URL документа |
| `checkbox_text` | Текст чекбокса или подтверждения |
| `accepted_at` | Дата и время принятия |
| `ip_address` | IP-адрес пользователя |
| `user_agent` | User-agent пользователя |
| `source` | Источник: registration, payment, profile, bot, api |

Примеры типов документов:

```text
terms
privacy
personal_data_consent
offer
payment_and_refund
subscription_rules
marketing_consent
cookie_policy
```

## 12. Финальная проверка

Перед завершением работы агент должен проверить:

- заполнена ли анкета проекта;
- определен ли тип проекта;
- определена ли модель B2B/B2C;
- выбран ли комплект документов по матрице;
- совпадает ли название сервиса во всех документах;
- совпадают ли реквизиты владельца;
- указана ли дата редакции документов;
- указана ли версия документов;
- нет ли противоречий между офертой, тарифами и правилами оплаты;
- указаны ли условия возврата;
- указаны ли платежные провайдеры;
- описаны ли персональные данные и цели обработки;
- описаны ли сроки хранения данных;
- описаны ли внешние сервисы и интеграции;
- описаны ли API-токены и ключи доступа;
- описаны ли банковские данные, если они используются;
- выделено ли отдельное согласие на обработку персональных данных;
- подготовлены ли тексты чекбоксов;
- подготовлены ли рекомендации по хранению согласий;
- подготовлен ли список плейсхолдеров;
- подготовлен ли список вопросов;
- подготовлен ли список юридических рисков;
- указано ли, что документы требуют юридической проверки.

## 13. Что агент должен вернуть в финальном ответе

В финальном ответе агент должен кратко указать:

1. Какие документы подготовлены.
2. Где они размещены.
3. Какие данные остались неизвестными.
4. Какие вопросы нужно уточнить у владельца проекта.
5. Какие вопросы нужно проверить с юристом.
6. Какие изменения нужно внести в интерфейс.
7. Какие согласия нужно хранить.

Обязательная фраза:

```text
Подготовленные документы являются черновиками и требуют юридической проверки перед публикацией.
```

---

## 14. Что агент не должен делать

Агент не должен:

- обещать юридическую пригодность документов;
- публиковать документы без подтверждения пользователя;
- выдумывать неизвестные данные;
- скрывать плейсхолдеры;
- копировать документы с чужих сайтов;
- игнорировать фактическую работу проекта;
- удалять важные ограничения ответственности;
- делать рекламное согласие обязательным без необходимости;
- изменять рабочий код проекта без отдельной задачи;
- смешивать все согласия в один общий чекбокс.
